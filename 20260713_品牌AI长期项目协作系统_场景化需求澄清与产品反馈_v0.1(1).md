## 先不要从技术架构开始，请先进入下面几个真实工作场景

这份反馈不是否定现有技术设计。

现有Wiki已经较完整地考虑了数据库、对象存储、权限、审计、并发、备份和多模型接口。问题在于：如果没有先理解品牌营销工作的真实过程，技术系统很容易把本来需要保留的开放性，当成需要立刻消除的“不确定性”；把讨论变成决定，把倾向变成红线，把希望完成的时间变成死线。

最终可能得到一个技术上严谨、业务上却持续误解人的系统。

以下场景，是当前产品必须首先解决的真实问题。

---

## 场景一：客户在会上说了“不要”，但这句话不一定是红线

一场鸿日项目讨论会上，可能出现这样的对话：

> “这个方向可以再试一下。”<br>
> “不要做得太像传统母婴产品。”<br>
> “安全还是要讲，但不要讲得太沉重。”<br>
> “月底前最好能看到一版。”

按照技术需求分析的习惯，系统很容易整理成：

- 已决定采用该方向；
- 红线：禁止母婴表达；
- 强制要求：必须突出安全；
- Deadline：本月底交付。

但在人类品牌会议里，这四条都可能是错的：

- “可以再试”通常只是一个选项获得继续探索的资格；
- “不要太像”可能是审美偏好或对当前版本的反馈，不等于永久禁令；
- “安全要讲”可能是当前倾向，仍需讨论以什么方式讲；
- “最好看到一版”可能是目标时间，不一定构成正式承诺。

### 正确的系统行为

系统应先判断会议模式是探索、评估、决策还是同步，再进行分类：

| 原话 | 建议类型 | 当前状态 | 能否直接改变项目状态 |
|---|---|---|---|
| 这个方向可以再试一下 | OPTION | working | 否 |
| 不要做得太像传统母婴产品 | VIEW / PREFERENCE | working | 否 |
| 安全还是要讲 | DIRECTION TENDENCY | preferred | 否 |
| 月底前最好看到一版 | TARGET DATE | tentative | 否 |

只有出现明确决定人、明确决定动词、适用范围和足够证据时，系统才能提出“这可能是一项决定”的建议；最终仍需人确认。

### 这个场景对应的产品能力

- 会议模式识别；
- 原话证据保留；
- 观点、假设、选项、倾向、决定和约束分类；
- 禁止AI自动升级状态；
- 人工确认队列；
- 时间性质区分：外部硬截止、内部目标、评审节点、暂定日期。

这不是普通会议摘要功能，而是“会议如何改变项目状态”的解释系统。

## 场景二：资料都在，但AI还是使用了错误版本

鸿日长期项目中同时存在：

- 最早的客户Brief；
- 两轮比稿需求；
- 两轮提案PDF；
- 两次真实述标录音；
- 后续客户会议；
- 品牌和产品客观资料；
- 市场、竞品和技术调研；
- 历史方案、废案和AI草稿；
- 当前正在推进的新方向。

Fox问AI：“基于现在的情况，下一轮应该怎么做？”

传统RAG或随机文件搜索可能搜到一份语义最接近、但已经过期的旧方案。AI随后引用旧判断、旧产品语言或已经被推翻的方向，而且表达得非常自信。

问题不是“文件没有上传”，而是系统不知道：

- 哪份资料是原始证据；
- 哪项结论现在仍然有效；
- 哪项只是某次提案里的候选；
- 哪项后来已经被取代；
- 当前任务应该先读什么，再按什么关系追溯。

### 正确的系统行为

AI开始工作时，不应先在所有文件中随机搜索，而应按固定路径读取：

1. 当前项目状态；
2. 当前阶段与本轮任务；
3. 已批准决定；
4. 仍然开放的问题；
5. 与本任务相关的证据和会议；
6. 必要时再打开完整原文；
7. 历史废案只在复盘、排重或风险检查时调用。

### 这个场景对应的产品能力

- 原始资料库；
- 文件与内容索引；
- 当前状态层；
- 事实与证据账；
- 决策与变更记录；
- 版本和有效期；
- `supersedes / supports / contradicts / derived_from` 等关系；
- 可以返回原始文件、会议、发言人和时间点的证据链接。

“把所有文件向量化”不能单独解决这个问题。检索相似，并不等于知道哪个内容当前有效。

## 场景三：品牌策略不是从需求直接计算出一个答案

Fox提出的问题可能是：

> 鸿日应该争夺哪一种“稳”？<br>
> 驾驶者的稳、后排乘坐的稳、车身底子的稳，哪一个是主叙事？<br>
> “高颜值”和“真稳”之间真正的冲突是什么？

这类问题不是字段缺失，也不是等待系统补齐验收标准。

品牌策略需要经历：

- 保留矛盾；
- 重新定义问题；
- 提出多个假设；
- 建立不同的战略领地；
- 比较每个选择获得什么、放弃什么；
- 经过人类判断后才收敛。

如果系统过早进入工程执行模式，它会立刻要求：

- 明确唯一目标；
- 锁定所有边界；
- 填满所有字段；
- 生成完整命名和文案；
- 对所有问题给出确定答案。

结果往往“结构完整”，但没有品牌策略的选择和创意锋芒。

### 正确的系统行为

系统必须存在两种不同的工作协议。

| 工作模式 | 适用场景 | AI应该做什么 | AI不得做什么 |
|---|---|---|---|
| 探索协议 | 研究、洞察、策略、创意方向 | 保留矛盾、提出假设、发展不同选择、揭示代价 | 急于形成唯一结论、直接锁定方向、用完整交付掩盖策略缺口 |
| 执行规格 | 已批准方向后的命名、文案、PPT、物料 | 严格服从事实、方向、格式和验收标准 | 擅自重写战略、把历史候选重新带回主线 |

工作模式的切换必须显式发生，不能由AI自行判断“信息已经足够，所以我替人收口”。

### 这个场景对应的产品能力

- 业务工作模式；
- 品牌策略师、创意总监、文案等运行时角色；
- 探索协议和执行规格；
- 品味与反例库；
- 人工选择闸门；
- 策略和创意质量评测。

这部分不是提示词装饰，而是产品核心行为。

## 场景四：加入一场新会议，不应该重写整个项目历史

项目已经运行数月，新加入一场会议记录。

错误做法是让AI重新总结全部历史，然后静默覆盖“当前状态”。这种方式容易发生：

- 新会议中的个人意见覆盖旧的正式决定；
- 同一句话被重复入库；
- 旧证据与新证据冲突却没有提示；
- AI总结的语言逐轮偏离原话；
- 人不知道这次到底改变了什么。

### 正确的系统行为

新会议进入后，系统只应生成一份“增量变化建议”：

- 新增了哪些事实候选；
- 出现了哪些新观点、假设和选项；
- 哪些内容可能与现有决定冲突；
- 哪些行动事项信息完整；
- 哪些项目状态建议修改；
- 每一项变化的原话依据是什么；
- 哪些必须由Fox确认。

确认后才写入当前状态；未确认内容继续停留在探索层。

### 这个场景对应的产品能力

- 增量摄取；
- 内容去重；
- 事件和变化记录；
- 冲突检测；
- 状态变更提案；
- 人工批准与驳回；
- 可回溯的状态历史。

事件账本在这里有价值，但它服务的是“项目发生了什么变化”，不是为了技术上拥有一个事件系统。

## 场景五：换一个模型，不应该重新解释整个鸿日项目

某一轮工作适合Codex做资料核对，另一轮可能更适合Claude做长文与品牌表达，某些中文任务也可能使用国内模型。

现在的痛点是，每换一个模型，Fox就要重新解释：

- 项目背景；
- 当前阶段；
- 哪些已经决定；
- 哪些仍然开放；
- 哪些资料可信；
- 这次到底要做什么。

### 正确的系统行为

不同模型不依赖各自聊天记忆，而是通过同一组稳定入口获得：

- 当前状态；
- 本轮任务包；
- 已确认决定；
- 相关证据；
- 开放问题；
- 当前工作模式；
- 对应的品牌角色和质量标准。

模型可以能力不同、表达不同，但不能各自拥有一套互相冲突的项目事实。

### 这个场景对应的产品能力

- 统一的本地API、MCP或CLI入口；
- 面向任务的上下文装配；
- 模型路由；
- 模型输出的统一证据要求；
- BrandBench与匿名人评。

多模型路由的重点不是“系统能调用多少模型”，而是“每个模型是否得到正确、最小、当前有效的上下文”。

## 场景六：未来可能团队使用，但现在必须先证明Fox本人真的提效

此前确实讨论过另一项需求：FoxWork中的大型项目需要多人协作，涉及NAS、文件同步、个人工作区、版本、权限和远程访问。

这项需求是真实的，但它与当前产品的第一验证目标不是同一件事。

### 当前主线

```text
/Users/fox/work
→ 独立、干净的品牌营销工作空间
→ Fox本人先使用
→ 鸿日项目先跑通
→ 验证AI理解、会议、状态、策略和多模型协作
```

### 未来可能的团队线

```text
个人试点有效
→ 判断哪些能力值得团队共用
→ 再设计服务器、账户、权限、并发、远程和灾备
```

团队服务器不是错误方向，但它是后续扩展条件，不是当前MVP的前置条件。

# 一、重新定义这个产品

这个产品不是：

- 普通项目管理软件；
- Worktile或Teambition的替代品；
- 通用企业知识库；
- 把所有文档向量化的RAG；
- 一套先建设完整基础设施、以后再找真实业务验证的平台。

这个产品首先是：

> 一个面向长期品牌营销项目的“项目状态与品牌认知协作层”：它让人和不同AI都知道什么是事实、什么只是讨论、现在做到哪里、为什么这样判断、本轮处于探索还是执行，以及每项结论可以回到哪条证据。

它需要同时服务两种完全不同的能力：

1. **可靠性**：事实、证据、状态、版本、权限和变更不能混乱；
2. **品牌工作能力**：能够保留开放性，形成策略选择，理解消费者语言，并接受人类品味判断。

只有第一项，会得到一个稳定的工程系统；只有第二项，会得到一个聪明但无法长期接续的聊天工具。当前产品必须把两者结合。

# 二、两条需求必须分开管理

| 需求线 | 当前问题 | 第一用户 | 当前优先级 | 技术形态 |
|---|---|---|---|---|
| A. 品牌AI长期项目协作 | AI失忆、会议误读、状态混乱、策略与文案工程化、多模型重复解释 | Fox | P0 | 本地优先、鸿日试点、轻量数据库与界面 |
| B. FoxWork团队文件协作 | 多人文件同步、NAS、版本、权限、远程和PPT合版 | 团队 | 独立后续议题 | 中央同步、NAS或服务器能力，另行确认 |

两条线未来可以共享某些底层能力，但不能在产品定义阶段直接合并，更不能把B的远期技术方案写成A的“已确认目标”。

# 三、每个系统模块到底服务什么场景

| 模块 | 关键作用 | 典型使用场景 | 不应被误解为 |
|---|---|---|---|
| 原始资料库 | 保留不可被AI改写的原始证据 | 打开客户Brief、提案PDF、录音和技术资料 | 把所有文件复制进一个大目录 |
| 内容索引 | 知道有哪些资料、主题、时间和来源 | AI先定位相关资料，再打开原文 | 一份人工维护且AI不读的MD清单 |
| 会议解释层 | 区分观点、假设、选项、决定、约束和行动 | 新会议进入后的分类与证据对齐 | 自动生成一份看似确定的会议摘要 |
| 当前状态层 | 告诉所有AI现在有效的项目状态 | 新会话冷启动、本轮任务开始 | 全部项目历史的长篇总结 |
| 证据与决策关系图谱 | 表达谁在何时基于什么提出、支持、反对或批准了什么 | 追溯结论、发现冲突、判断内容是否过期 | 只把实体和关键词连成漂亮脑图 |
| 状态变更提案 | 让AI提出改变，但不能自行提交 | 新会议、新资料、新反馈后的增量更新 | AI自动维护并覆盖项目真相 |
| 品牌Agent宪法 | 规定AI以什么业务角色、模式和品味工作 | 策略诊断、创意探索、命名、文案 | “你是一位国际4A专家”式角色标签 |
| 任务上下文装配 | 针对本轮任务提供最小、相关、当前有效的信息 | Codex和Claude开始一项具体工作 | 把全部资料塞进上下文 |
| 人类操作界面 | 让Fox查看、确认、驳回和打开证据 | 查看当前状态、待确认变化和任务 | 重新开发一套通用项目管理平台 |
| 多模型接口 | 让不同AI读取同一项目真相 | 按任务切换Codex、Claude或其他模型 | 同时调用更多模型就自然变好 |
| BrandBench | 判断系统是否真的减少错误并提升品牌质量 | 模型升级、流程调整和版本验收 | 只测接口、响应时间和代码覆盖率 |

# 四、图谱应该表达什么

这里需要的“图谱”，不是只给人看的脑图，也不是把所有名词自动连接。

第一阶段应优先表达以下业务对象：

- 项目阶段；
- 会议；
- 发言人和决定人；
- 原始资料；
- 事实；
- 观点；
- 假设；
- 选项；
- 决定；
- 约束；
- 开放问题；
- 行动；
- 战略领地；
- 方案与交付物。

需要表达的关键关系包括：

- `提出于`；
- `来源于`；
- `支持`；
- `反对`；
- `与之冲突`；
- `适用于`；
- `被谁批准`；
- `替代了`；
- `依赖于`；
- `回答了哪个问题`；
- `仍待确认`。

第一阶段不一定需要独立图数据库。SQLite或其他轻量结构化存储也可以表达这些关系。是否升级到图数据库，应由真实查询复杂度决定，而不是由“知识图谱”这个名词决定。

# 五、AGENTS与运行时品牌角色必须拆开

现有Wiki中的开发仓库 `AGENTS` 主要规定数据库、API、存储、测试和部署。如果它服务的是软件开发者，这些内容本身没有问题。

问题是：开发规则不能代替产品运行时的品牌业务规则。

建议至少拆成四类文件：

| 文件层 | 负责什么 |
|---|---|
| 开发仓库AGENTS | 代码架构、测试、接口、数据安全和开发约束 |
| 品牌Agent宪法 | 品牌策略角色、创意角色、证据纪律、品味与语言原则 |
| 工作模式协议 | 探索、评估、决策、执行各自允许和禁止的行为 |
| 项目级Agent规则 | 鸿日当前目标、已确认方向、开放问题、事实边界和本轮非目标 |

特别需要明确：

> “AI是品牌策略师”不是一句角色扮演提示，而是一组可验证行为：它能否保留矛盾、提出不同战略选择、说明代价、理解消费者语言，并在未经人类批准时不擅自收口。

# 七、建议的第一阶段MVP

## Phase 0：目标和测试先行

产出：

- 一页产品边界；
- 鸿日真实资料范围；
- 10—20个黄金测试案例；
- 事实、观点、决定、约束、行动的分类标准；
- 探索与执行模式标准；
- 第一版品牌质量评分表。

这一阶段的目的，是在写系统前先定义“什么叫理解正确”。

## Phase 1：本地鸿日原型

建议范围：

- 本地优先；
- 单用户；
- 原始资料只读保存；
- 轻量数据库或索引；
- 会议解析和原话证据；
- 当前状态与状态变更提案；
- 关系查询；
- 品牌Agent运行规则；
- Codex和Claude可调用；
- 一个简单的本地查看与确认界面。

第一阶段不要求：

- 团队账户；
- 生产服务器；
- PostgreSQL；
- S3；
- 完整PWA；
- 企业权限体系；
- 高可用与灾备；
- 所有第三方平台集成。

## Phase 2：真实工作验证

用鸿日正在发生的工作连续验证：

- 新会议进入；
- 当前状态变化；
- 策略诊断；
- 方案发展；
- 文案与命名；
- 多模型切换；
- 证据回查；
- 一次错误后的规则修订。

系统不是“功能完成”就算通过，而是Fox真实使用后确实减少解释、减少混乱并提升产出质量。

## Phase 3：再决定是否团队服务器化

只有Phase 2证明有效后，再回答：

- 是否需要多人同时使用；
- 哪些数据必须放服务器；
- 是否需要PostgreSQL和对象存储；
- 是否需要账户、权限、审计和并发；
- 是否与FoxWork团队协作需求合并；
- 哪些能力仍保持本地优先。

# 八、必须通过的真实验收用例

| 用例 | 操作 | 通过标准 |
|---|---|---|
| 冷启动 | 一个没有读过旧聊天的新AI进入鸿日 | 能准确说明当前阶段、目标、决定、开放问题和本轮任务，并给出来源 |
| 会议解释 | 输入一场包含讨论、偏好、暂定时间的会议 | 不把VIEW、OPTION、HYPOTHESIS升级成DECISION或CONSTRAINT，不把目标时间写成Deadline |
| 证据回查 | 询问“为什么说这个方向已确认” | 返回决定人、原话、会议、时间和适用范围；证据不足时明确说未确认 |
| 增量更新 | 新加入一场会议 | 只提出变化建议，不静默覆盖旧状态；冲突单列 |
| 策略探索 | 询问鸿日应争夺哪一种“稳” | 给出真正不同的战略选择及代价，不直接跳到命名和口号 |
| 执行落地 | 人工批准一个方向后要求写语言系统 | 严格服从已批准方向，不重新发明战略，不带回废案 |
| 模型切换 | 同一任务分别交给Codex和Claude | 两者读取到相同的当前事实、决定和证据，差异主要来自能力和风格 |
| 品牌质量 | 对策略与文案候选进行匿名评审 | 在战略锋利度、自然中文、消费者真实、记忆性和产品咬合度上达到设定基线 |

### 一票否决错误

- 虚构产品事实；
- 把讨论升级成决定；
- 把暂定日期写成死线；
- 把过期方案当成当前方向；
- 重要结论不能回到证据；
- 未经人确认自动改变项目状态；
- 在探索模式下强行制造唯一答案。

# 最后的共同判断

这次需要修正的不是“技术做得不够完整”，而是产品验证顺序：

> 先证明系统能够正确理解品牌项目，再证明它能够稳定存储这些理解；先证明Fox本人真正提效，再决定如何让团队共同使用。

品牌项目中的不确定性，不全部是需要被消灭的噪音。

有些是事实不足，需要验证；有些是观点冲突，需要保留；有些是战略选择，需要比较；有些才是已经确认的决定，需要锁定。

这个系统真正的价值，就是把这些不同性质的内容分清楚，并让人始终拥有最终判断权。
