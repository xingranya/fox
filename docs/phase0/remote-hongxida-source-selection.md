# 鸿喜达远程资料筛选结论

> 日期：2026-07-21  
> 范围：用户授权的远程鸿喜达项目目录  
> 原则：只读筛选、当前状态优先、最小必要读取、敏感原文不提交仓库

## 筛选结论

远程资料根约 14GB，包含当前生产、治理、历史归档、研究、素材和 AI 工作区。Phase 0 不应把全部文件直接塞入 AI 上下文，而应先通过项目已有 SSOT 和五态治理装配当前 Task Packet。

本轮选出 9 份当前来源并完成 SHA-256 核验：

| 来源角色 | 用途 | 当前权威边界 |
|:---|:---|:---|
| 客户级协作规则 | 确认项目组合、品牌名称、保密和当前优先级 | 规则权威，不承载具体产品事实 |
| 项目级协作规则 | 确认五态、资料优先级和当前必读顺序 | 规则权威，不替代客户/研发证据 |
| 项目总控 | 当前阶段、项目关系、主线和关键闸门 | 项目控制 SSOT |
| 当前焦点 | 当前 P0、可开工、不可交付和 Fox 待拍 | 当前状态投影 |
| 资源清单 | 当前真源、工作源、历史覆盖和缺失资源 | 来源路由与使用边界 |
| 决策日志 | Fox 确认、客户方向、候选、待决和禁用 | 人工决策与状态形成证据 |
| 战役看板 | 7·29—30 战役边界、Gate、工作包和执行快照 | MD 执行入口；实时 Owner/截止仍以唯一 Base 为准 |
| 产品二次包装状态说明 | 当前工作源、创作与发布边界 | 工作源规则 |
| 产品二次包装总纲 v0.1 | 总价值、三答案、六模块、三 SKU 和证据系统候选 | `proposal_candidate`，不是对客定稿 |

完整远程路径、文件哈希、匿名映射和评审候选只保存在被 Git 忽略的 `.work/phase0/`。

## 当前项目状态纠偏

- 当前最高优先级不是旧半年度本地生活规划，而是 `EVENT-0730` 经销商会与新品发布战役。
- `HIT-L206` 长期爆品建设保留，但会前只推进直接解锁战役的事实、内容和资产。
- 本地生活和管理者个人 IP 是独立项目线，不得混入当前战役 Task Packet。
- 当前已经从 AI 并行生产转入人类选择、客户回填、专业完稿、资源协调与交付验收阶段。
- “文件存在”“客户讨论”“AI 候选”“Fox 确认”必须分开；任何文件名中的“定稿”均不能代替状态证据。

## 五态映射

远程项目现有五态与 Brand Project OS 的映射如下：

| 远程项目状态 | Brand Project OS 层级 | 处理规则 |
|:---|:---|:---|
| `confirmed_by_fox` | 已批准事件/正式状态候选输入 | 仍须保留范围、证据和版本 |
| `client_direction` | `TENDENCY`、`DECISION_CANDIDATE` 或 `CONSTRAINT_CANDIDATE` | 不自动批准，等待回签或 Fox 确认 |
| `proposal_candidate` | 工作层 `OPTION` 或 Proposal | 只能比较、修改、驳回或人工批准 |
| `pending_fact_or_decision` | `OPEN`、待证据或待人工动作 | 不补全、不承诺 |
| `superseded_or_forbidden` | 历史/禁用层 | 默认不进入当前上下文，只用于排重与风险检查 |

## 进入新评审包的业务范围

新 BrandBench 任务聚焦鸿日小DUN产品二次包装，要求候选同时处理：

1. 一个总价值；
2. 三个消费者答案；
3. 六个产品证明模块；
4. Pro / Ultra / Ultra+ 三个 SKU 的边界；
5. “创作开放、发布核验”的 Claim 规则；
6. Fox 当前需要完成的选择。

旧比稿、旧 IP、旧 SKU、旧传播日历、归档稿、AI 概念图和未签发强技术结论均未进入评审。
