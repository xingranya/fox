# FoxWork 与 AI 访问 SPEC

> 员工客户端：FoxWork，唯一安装；FoxWork、Den 员工页和后台管理员网页全部使用简体中文
> 账号与 AI 控制面：公司自托管 OpenWork Den  
> 品牌业务后端：Brand Project OS Service  
> 当前任务：F3.3；客户端主改造在 F3.4-F3.13

## 员工实际流程

```text
安装 FoxWork
  -> 注册或登录公司 Den 账号
  -> 回到 FoxWork，进入唯一公司组织
  -> 自动同步可用项目、模型、MCP、Skills 和桌面策略
  -> 拖入资料，查看处理进度和分析结果
  -> 向 AI 提问或发起任务
  -> 查看依据与差异
  -> 有权限员工逐项批准、修改或驳回 Proposal
```

员工不安装第二个 Brand Project OS 客户端，不维护第二套账号，不手填 MCP Token、模型 API Key 或 Brand OS 地址。Den Web 只承担注册、登录、授权和必要管理；日常项目工作回到 FoxWork 完成。

## 入口与职责

| 入口 | 使用者 | 能力 | 明确禁止 |
|:---|:---|:---|:---|
| FoxWork | 员工 | 公司工作区、项目、资料、会议、证据、Proposal、AI 工作、流程和诊断 | 从 Session 推导正式状态，直连数据库或对象存储 |
| Den 管理面 | 公司管理员 | 成员、团队、远程工作区、Worker、MCP、Skills、插件、共享模型和桌面策略 | 管理项目正式事实或代替业务批准 |
| 本机桥接 | 当前设备上的员工 | 经授权的文件、终端和桌面动作 | 服务器绕过授权、跨目录扫描或产生业务批准 |
| Brand OS API | FoxWork | 查询、上传、处理状态、证据、Proposal 和人工确认 | 接受 Den Session Token 直通或服务账号人工路由 |
| Brand OS MCP | Codex、Claude、Den Agent、Dify | 按项目 Scope 读取、回源、Task Packet 和创建 Proposal | 跨项目、批准、成员管理、任意 SQL 或任意文件 |
| CLI / stdio MCP | 运维和本机 Agent | 初始化、迁移、校验、诊断及同一白名单用例 | 非交互入口声明为人工身份 |
| Skills | FoxWork 与 Agent | 版本化工作法、工具顺序、输入输出和失败处理 | 保存实时事实、原件、密钥或私有聊天记忆 |

## 信息架构

FoxWork 首屏是公司工作区，不是营销页，也不是单纯聊天首页。界面保持紧凑、可扫描，普通员工看业务语言，管理员再进入设置和诊断。

| 一级入口 | 员工要解决的问题 | 核心内容 |
|:---|:---|:---|
| 工作区 | 我属于哪个公司，能处理哪些项目 | 当前组织、项目列表、同步状态和最近工作 |
| 当前 | 项目做到哪里，什么已经确认 | 阶段、目标、决定、约束、开放问题、行动和状态版本 |
| 资料 | AI 读了什么，处理到哪一步 | 原件版本、上传/检查/解析状态、来源、缺口和打开原文 |
| 会议 | 这次会议可能改变了什么 | 会议模式、原话、分类、重复、冲突和增量 Proposal |
| 待确认 | 哪些变化需要人判断 | 旧新差异、原话、证据、影响、冲突和逐项操作 |
| AI 工作 | AI 领取了什么、做了什么 | Task Packet、模型、工具请求、过程、Artifact 和采用结果 |
| 流程 | 自动化现在进行到哪一步 | Dify/FlowLong/Worker 状态、失败、重试和人工待办 |
| 公司能力 | 我能使用哪些模型、MCP 和 Skills | Den 下发的能力、权限说明和不可用原因 |
| 设置 | 账号、设备和连接是否正常 | 登录会话、公司入口、客户端版本、更新和必要诊断 |

旧“团队连接”页删除。它的账号、登录、公司连接和能力配置整合进“工作区”“公司能力”和“设置”，不再出现两套相似入口。

## 中文文案要求

- 菜单、按钮、注册登录、提示、错误、通知、安装、更新、深链、Den 员工页面和后台管理员网页全部使用自然简体中文。
- 不提供语言切换；缺失文案不能回退为英文、上游品牌、内部键名或开发占位。
- 普通员工界面不出现 `OIDC`、`audience`、`scope`、`RLS`、`Outbox`、`Provider`、`MCP Token` 等实现词。
- 错误必须回答“发生了什么、当前能做什么、是否会丢数据”。例如显示“公司登录已过期，请重新登录”，而不是 `fresh_auth_required`。
- 已批准、工作中、待确认、已失效使用文字标签与分区，不只靠颜色。

## 登录与公司连接

1. 首次启动读取公司预置的 Den Web 地址；允许管理员为受控内网显式配置 HTTP 入口。
2. FoxWork 打开 Den 自助注册或登录页，完成桌面交接并回到应用；注册后只能进入唯一公司组织。
3. 客户端显示公司名称、员工身份和登录状态，再获取 Brand OS 专用短期令牌；首次访问时按可信 `(issuer, subject)` 建立内部身份映射，不要求管理员预建 Brand OS 账号。
4. 客户端按 Den 组织/团队与 Brand OS 项目映射加载项目，不允许客户端自己拼接项目 ID。
5. 登出清除 Den 与 Brand OS 的可恢复令牌密文、缓存的敏感数据和本机运行授权。
6. 账号或成员撤权后，旧会话、模型、MCP、Skills 和项目访问都必须失效。

Den 和 Brand OS 使用不同 audience 的短期令牌，但员工只看到一次登录。首次身份建档默认没有项目权限或业务审批权；邮箱不能用于自动合并或重新绑定身份。密文只保存在系统钥匙串，不写 `localStorage`、URL、日志、Prompt、Skill、项目目录或 OpenWork SQLite。

## 资料与分析

- 支持图片、视频、录音、PPT、Office 和 PDF 的选择、拖入、分片上传、断点续传、取消和重试。
- 上传前显示文件名、大小、项目和保密提示；上传后显示检查、排队、处理中、完成、失败或已取消。
- 同名异内容保存为不同版本，不静默覆盖。
- 分析结果必须带原件版本和定位：图片区域、页码、幻灯片号、表格/段落或音视频时间码。
- 解析、OCR、转写或模型失败不影响原件 ACTIVE 状态；可重新处理，不重复上传原件。
- AI 结果先成为 Artifact 或 Proposal。需要改变正式状态时，必须进入“待确认”。

## 两类确认

| 项目 | 业务确认 | Tool Permission |
|:---|:---|:---|
| 目的 | 接受、修改或驳回正式状态变化 | 允许本次 Agent 使用文件、终端、网络或工具 |
| 身份 | Den 登录后的交互式员工 + Brand OS 项目权限 | 当前员工或受限运行策略 |
| 处理位置 | Brand OS 人工确认用例 | OpenWork/OpenCode 运行层 |
| 结果 | 追加审批与领域事件，更新投影 | 只影响本次运行 |
| 文案 | 批准、修改后批准、驳回、暂缓 | 本次允许、拒绝、限范围允许 |

两类动作不得共用按钮、Schema、路由或自动通过设置。AI 页面里的“采用”只创建 Proposal；真正改变正式状态仍在“待确认”中完成。

## Task Packet 与模型

每次 Agent 运行绑定不可变 Task Packet：

- L0：角色、模式、目标、交付、非目标和输出 Schema；
- L1：当前阶段、批准事实/决定/约束、开放问题、行动和状态版本；
- L2：本任务相关证据、会议和关系；
- L3：按需打开的原始内容；
- L4：只在复盘、排重、冲突和风险检查时读取的历史与废案。

FoxWork 显示 Packet 版本、证据范围、数据外发范围和模型。Den 负责给员工下发可用模型及密钥，Brand OS 只保存模型 ID、Packet 哈希、运行版本和结果引用，不读取模型 API Key。切换模型必须复用同一 Packet，不能让每个模型靠自己的聊天记忆维护项目事实。

## MCP 与 Skills

Brand OS MCP 的首批服务器能力沿用本地白名单：

- `project_get_state`
- `task_get_packet`
- `evidence_get`
- `decision_list`、`open_question_list`
- `proposal_create`、`proposal_get`
- `system_doctor`、`project_verify`

远程入口增加 OAuth、项目 Scope、成员/团队授权、撤权、限流和审计，不增加批准、直接 SQL、成员管理、硬删除原件、强制切换工作模式或任意本机文件能力。

Den 负责能力发现和下发。Skill 有稳定 ID、版本、签名、兼容范围、变更记录和回滚版本；运行时事实始终从 Brand OS API/MCP 获取。

## Dify 与可选组件

| 组件 | 任务 | 接入条件 | 退出方式 |
|:---|:---|:---|:---|
| Dify | F3.14 | 外发清单、服务身份、超时、取消、回调幂等和审计通过 | 切回直接 Worker/NoOp |
| Zvec | F3.15 | 相对 PostgreSQL FTS 有可测收益，权限回源完整 | 删除索引并回退 FTS |
| Open Notebook | F3.16 | 解析质量、来源定位、失败隔离和许可通过 | 停用适配器，保留原件与映射 |
| Nubase | F3.17 | 记忆污染、删除和项目隔离通过 | 清空记忆并回退 Task Packet |
| FlowLong | F3.18 | 许可、回调、撤回、超时和越权测试通过 | 使用 Brand OS 内置待确认队列 |

自托管 Dify 中的外部模型、插件和 HTTP 节点仍属于数据外发。任何组件都不能批准正式状态，且全部禁用时核心业务旅程仍应完成。

## 离线与故障

- FoxWork 缓存必须显示状态版本、服务器水位和同步时间。
- Den 不可用时停止新登录、组织和能力变更；已有 Brand OS 短期会话只运行到明确过期。
- Brand OS 不可用时只允许查看带水位缓存、编辑草稿和准备 Proposal；批准、驳回和正式提交禁用。
- 恢复联网后先刷新身份、权限和当前版本，再把草稿生成为新差异；不自动合并决定。
- OpenCode/Sidecar 故障不影响状态、证据和资料读取；AI 工作暂停并显示可操作错误。
- Dify、Zvec、Open Notebook、Nubase 或 FlowLong 故障不影响核心 API 就绪。

## 客户端安全

- Electron 启用 Sandbox、`contextIsolation`、最小 preload 和 IPC 允许列表。
- Renderer 不直接获得 Node、数据库、对象存储、系统钥匙串或任意文件系统能力。
- 本机路径经过工作区允许列表、真实路径、符号链接和目录边界复核。
- 外部导航、下载、深链、协议、Den 入口和网络出口使用允许列表。
- 服务器不能反向调用员工电脑；本机桥接只接受 FoxWork 主进程签发的短期授权。
- 无公司配置时不得回退上游公共 Den、遥测、模型目录、更新或云服务。

## 验收

1. 员工安装一个 FoxWork 就能自助注册登录、进入唯一公司组织和获授权远程工作区、选择项目、使用模型/MCP/Skills、上传资料并处理 Proposal。
2. FoxWork 中不存在第二套 Brand OS 密码、登录码或旧“团队连接”入口。
3. FoxWork、Den 员工页和后台管理员网页全部为自然简体中文，产物扫描无英文回退、上游品牌和内部键名。
4. 两个客户端并发确认同一 Proposal 时，只有一个版本成功，另一端看到可理解差异。
5. 账号、团队、项目或能力撤权后，旧客户端、REST、MCP 搜索/执行和模型连接全部拒绝。
6. Codex、Claude、Den Agent 和 Dify 读取同一 Task Packet 时，正式事实、决定和证据一致。
7. 服务账号、MCP、Skill、Dify、FlowLong 和 Tool Permission 均无法批准正式状态。
8. 服务器不能绕过 FoxWork 访问员工电脑；本机桥接每次按工具和路径授权。
9. 清除 OpenWork/Den Session、缓存和外部组件派生数据后，正式状态和证据仍完整。
10. 普通员工可以理解错误和下一步，不需要数据库、协议或 AI 工程知识。
