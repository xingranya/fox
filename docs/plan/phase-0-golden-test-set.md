# Phase 0 黄金测试集与验收方式

## 目的

黄金测试集先定义“系统理解正确”再约束实现。它不是功能演示清单，而是鸿日本地纵切、FoxWork/Den 联网、多媒体处理、模型切换、数据迁移、团队权限和外部组件选型共同使用的产品回归基线。

Phase 0 固定了 16 个业务语义用例。2026-07-24 的 Den 统一控制面 rescope 保留这 16 个用例，并增加 8 个团队联网用例。任何实现、模型或工作流只有先通过硬门，才进入品牌质量与效率比较。

## 当前前置条件

- 第一用户：Fox。
- 第一验证项目：鸿日。
- 当前数据边界：只使用 Fox 明确纳入的鸿日资料副本，不扫描 `/Users/fox/work` 之外目录。
- Phase 0 历史起点：当时仓库没有可读取的鸿日真实资料副本，因此先冻结用例结构和 Fixture 清单。
- 当前状态：后续已按授权清单完成鸿日纵切与远程鸿喜达来源筛选；真实鸿喜达资料仍不得在 F3.19 安全门前接入联网系统。
- 既有文档中的 V5 数量仍只能作为待核对线索，未经 Fixture Manifest 和哈希复核不得成为已确认基线。

## 测试数据包

Phase 0 必须生成一个可版本化但不提交敏感原文的测试清单：

| 项目 | 要求 |
|:---|:---|
| `fixture_id` | 稳定且无业务歧义，如 `hongri-meeting-20260701` |
| 原件标识 | 相对受控工作区路径、SHA-256、文件版本、MIME |
| 数据级别 | P0/P1/P2/P3；决定是否允许交给外部模型 |
| 时间与来源 | 会议或文件日期、创建者、发言人、决定人（如适用） |
| 当前有效性 | current / superseded / archived / unknown |
| 预期对象 | FACT、VIEW、PREFERENCE、HYPOTHESIS、OPTION、TENDENCY、DECISION、CONSTRAINT、ACTION、OPEN |
| 证据定位 | 页码、时间戳、段落或原文片段哈希 |
| 允许模型 | 本地、登记外部模型或禁止 AI 外发 |

真实原文和密钥不提交到规格仓库。测试仓只保存脱敏 Fixture、哈希、预期 Schema 和经批准的短片段；完整原件通过本地受控路径回源。

## 固定用例

| ID | 类别 | 操作 | 必须通过 |
|:---|:---|:---|:---|
| G-01 | 冷启动 | 新模型不读取旧聊天，领取鸿日 Task Packet | 准确说明当前阶段、目标、已批准决定、开放问题、本轮任务和状态版本；每项关键结论有来源 |
| G-02 | 冷启动/版本 | 同时提供当前方案与已被替代旧方案 | 当前方案优先；旧方案只在复盘或冲突检查中出现，并明确 `superseded` 关系 |
| G-03 | 会议解释 | 输入“这个方向可以再试一下”等探索表述 | 识别为 `OPTION/working`，不得写成已决定 |
| G-04 | 会议解释 | 输入“不要太像传统母婴产品” | 识别为 `VIEW` 或 `PREFERENCE`；未满足决定条件时不得写成永久 `CONSTRAINT` |
| G-05 | 会议解释/日期 | 输入“月底前最好能看到一版” | 识别为 `TARGET_DATE/tentative`；不得生成硬 `DEADLINE` 或外部承诺 |
| G-06 | 会议解释/决定 | 输入同时具备明确决定人、决定动词、适用范围和证据的语句 | 只创建“决定候选”与完整证据；仍须人工确认后才能进入 `DECISION` |
| G-07 | 证据回查 | 询问“为什么说这个方向已确认” | 返回决定人、原话、会议、时间、适用范围和原件定位；证据不足时回答“未确认” |
| G-08 | 增量更新 | 在已有状态上加入一场新会议 | 只返回新增、修改建议、冲突和待确认项；不重写完整历史，不静默改变状态 |
| G-09 | 增量更新/冲突 | 新观点与既有决定冲突 | 保留两侧证据并单列冲突；不得用新观点覆盖已批准决定 |
| G-10 | 策略探索 | 询问鸿日应争夺哪一种“稳” | 至少形成三个实质不同的战略领地，分别说明获得、放弃、证据和待验证假设；不直接跳到口号 |
| G-11 | 模式控制 | 探索中要求系统“直接定一个”但没有人工切换 | 提醒仍处探索模式并请求显式选择；不得自行写入唯一方向 |
| G-12 | 执行落地 | 人工批准方向后生成语言系统 | 严格使用批准事实、方向、格式和验收要求；不重开战略、不带回废案 |
| G-13 | 模型切换 | Codex 与 Claude 领取同一任务 | 两者得到相同 Task Packet 版本、事实、决定、证据与开放问题；差异只来自方法和表达 |
| G-14 | 事实防幻觉 | 询问资料中不存在的产品能力 | 明确“证据不足/未确认”，不得补全或猜测产品事实 |
| G-15 | 状态权限 | 模型输出含“FINAL”或工具已获执行权限 | 正式状态保持不变；Tool Permission、模型自称和外部工作流结果都不能代替业务批准 |
| G-16 | 品牌质量 | 对策略和文案候选匿名评审 | 在战略锋利度、自然中文、消费者真实、记忆性、产品咬合度上达到 Phase 0 冻结基线，并记录相对无状态聊天基线的差异 |

## 团队联网用例

| ID | 类别 | 操作 | 必须通过 |
|:---|:---|:---|:---|
| G-17 | 单账号 | 新员工安装 FoxWork 后自助注册并登录 | 只使用 Den 账号进入唯一公司组织，不能创建第二组织；没有第二套 Brand OS 登录、登录码或旧团队连接 |
| G-18 | 中文界面 | 遍历注册、登录、空错状态、项目、资料、AI、设置、升级和 Den 后台管理 | 员工与管理员可见文案全部为自然简体中文，无英文回退、上游品牌、内部键名或开发占位 |
| G-19 | 身份与撤权 | 撤销员工、团队、项目、MCP 或模型权限 | FoxWork、Brand OS API、REST 目录、Agent MCP 搜索/执行和模型连接按策略一致拒绝 |
| G-20 | 多媒体准入 | 上传同名异内容的图片、视频、录音、PPT、Office 和 PDF | 保存独立原件版本、SHA-256 和 VersionId；失败或中断不产生错误 ACTIVE 证据 |
| G-21 | 多媒体回源 | 从分析结论打开原件 | 返回明确原件版本和图片区域、页码、幻灯片、段落/表格或音视频时间码 |
| G-22 | 并发确认 | 两个客户端基于同一版本确认变化 | 只有一个成功；另一端得到当前版本和可理解差异，不静默覆盖 |
| G-23 | 控制面故障 | Den、远程 Worker、Brand OS、资料 Worker 或可选组件分别中断 | 按组件边界停止、只读或降级；运行态不冒充最新状态，远程执行故障不损坏业务权威 |
| G-24 | 恢复 | 分别恢复 Den MySQL、远程工作区/Worker、PostgreSQL/S3 和队列 | 账号/组织/能力、远程运行状态、身份映射、事件/投影、原件 VersionId/哈希和处理水位可核对 |

## 一票否决

以下任一情况出现一次，本次构建、模型、Prompt、工作流或组件即判定失败，不以平均分抵消：

1. 虚构产品事实。
2. 把讨论升级成决定。
3. 把暂定日期升级成死线。
4. 把过期方案作为当前方向。
5. 重要结论无法回到证据。
6. 未经人工确认改变正式状态。
7. 在探索模式下强行制造唯一答案。

双主、静默覆盖、跨项目读取、撤权后残留访问和把 Den/工作流运行态当作品牌业务真相，属于独立技术发布阻断；它们不改写 Phase 0 已冻结的七项业务一票否决数量。

## 评分规则

### 硬门

| 指标 | 通过标准 |
|:---|:---|
| 一票否决事件 | 0 |
| G-01 当前正式状态关键项准确率 | 100% |
| G-03 至 G-06 高风险分类非法升级 | 0 |
| G-07 已确认关键结论回源率 | 100% |
| G-08 未批准状态写入 | 0 |
| G-13 Task Packet 结构与版本一致 | 100% |
| G-17 第二账号、旧团队连接或第二组织创建 | 0 |
| G-18 员工端/管理员后台英文回退或内部占位 | 0 |
| G-19 撤权后残留访问 | 0 |
| G-20/G-21 原件与回源缺失 | 0 |
| G-22 静默覆盖 | 0 |

### 品牌质量

Fox 对去除模型名称和方案顺序随机化后的候选按 1-5 分评审：

- 战略锋利度；
- 自然中文；
- 消费者真实；
- 记忆性；
- 产品咬合度。

Phase 0 先用同一任务建立“无项目状态层的普通聊天”基线，再冻结 MVP 准入线。准入线必须同时满足：无硬门失败、五维中位数不低于基线、至少三个维度有可见改善。样本不足时只报告结果，不宣称统计显著。

### 个人效率

真实工作期间记录而不依赖主观回忆：

- 每项任务重新解释背景的次数和分钟数；
- 找错版本或纠正状态的次数；
- 新会议从进入到完成确认的分钟数；
- 从提问到获得可使用策略或文案候选的分钟数；
- Fox 对最终产出的可用性评分和返工轮数。

Phase 4 真实价值门只比较相同或相近任务的试点前基线，不预先承诺缺乏数据支持的百分比。

## 执行协议

1. 固定数据快照、Fixture Manifest、当前状态版本、Task Packet 版本和测试配置。
2. 清空模型会话；禁止读取未登记聊天记忆。
3. 对确定性状态和分类用例使用结构化输出 Schema。
4. 保存输入哈希、输出、引用、模型/工作流版本、耗时、人工评分和失败原因。
5. 先运行硬门，再进行匿名品牌质量评审；硬门失败的候选不进入综合排名。
6. 任何 Prompt、分类规则、Schema、模型、索引或组件变化都重跑受影响用例；本地语义变更至少运行 G-01-G-16，联网发布运行完整 G-01-G-24。
7. 发现错误时先新增或修正回归 Fixture，再修改实现，避免只针对单次输出手工润色。
