# 安全与验收 SPEC

> 范围：FoxWork、OpenWork Den、Brand Project OS Service、数据存储、MCP/Skills/模型、Worker、Dify 与可选组件  
> 当前任务：F3.3  
> 原则：员工一次登录；权限最小；AI 只提 Proposal；重要结论可回源；故障不伪成功

## 安全目标

1. 员工只维护一套 Den 账号，Brand OS 不建立第二套密码体系。
2. Den Session、Brand OS 资源令牌、MCP 服务身份和人工业务身份严格分开。
3. 原件、正式事件、审批和当前投影不被 Session、模型摘要、索引或工作流覆盖。
4. 任何 AI、MCP、Skill、Agent、Dify、FlowLong、Den 管理动作或服务账号都不能批准业务状态。
5. 新资料和会议只生成增量 Proposal；未经人工确认不改变正式状态。
6. 项目访问先经过应用角色、动作、保密级别授权，再由 RLS 纵深防御。
7. 本机文件、终端和桌面只在 FoxWork 当前设备上按员工授权访问，服务器不能反向绕过。
8. 外部模型、插件、转录、Embedding 和 HTTP 节点的数据范围可枚举、最小化、撤销和审计。
9. Den、Brand OS、MySQL、PostgreSQL、S3 和客户端都能独立备份、恢复和撤权。
10. FoxWork、Den 员工页面和 Den 后台管理员网页全部为自然简体中文，安全错误不泄露内部结构或秘密。

## 信任边界

| 区域 | 允许 | 禁止 | 主要控制 |
|:---|:---|:---|:---|
| FoxWork Renderer | 展示业务、发起受控调用 | Node、系统钥匙串、数据库、任意文件系统 | Sandbox、`contextIsolation`、最小 preload、IPC Schema |
| FoxWork 主进程/本机桥接 | 系统钥匙串、用户选定路径、受控终端 | 跨目录扫描、局域网无认证监听、静默上传 | 路径复核、短期授权、来源/网络允许列表 |
| Den Web/API | 登录、组织、团队、能力目录和策略 | 保存品牌业务真相、自动授予项目审批权 | 新鲜登录、组织规则、成员/团队授权、审计 |
| Brand OS API | 业务查询、上传、Proposal 和人工路由 | 接受 Den Session 直通、服务账号调用人工路由 | 第一方资源令牌、项目 RBAC、RLS、路由隔离 |
| PostgreSQL/S3 | 正式事件、审批、投影、原件 | 客户端/Agent 直连、同名覆盖、双主 | 最小角色、事务、版本、哈希、VersionId、墓碑 |
| Agent/MCP/Skills | 读取最小 Task Packet、使用白名单工具、创建 Proposal | 业务批准、跨项目、永久凭据、任意本机文件 | 短期 Scope、封闭 Schema、执行时鉴权、审计 |
| 外部工作流/模型 | 经授权的最小输入 | 默认接收敏感原文、密钥或完整项目镜像 | 数据分级、提供商清单、任务授权、NoOp 退出 |

## 身份与会话

- 登录使用 Authorization Code + S256 PKCE、一次性 state/nonce 和受控回调。
- Brand OS 只接受面向自身 audience 的短期令牌，按 `(issuer, subject)` 绑定员工；可信 Den 成员可在首次访问时建立内部身份映射，但默认没有项目权限，邮箱不能触发合并或重新绑定。
- 邮箱、显示名、Den 组织管理员或模型会话都不能作为自动创建、合并、重新绑定或提升 Brand OS 身份的依据；首次建档只信任已校验的 `(issuer, subject)`、唯一公司组织和有效成员关系。
- Den Session Token 不直接访问 Brand OS；MCP 服务令牌不冒充发起操作的员工。
- 令牌和设备密钥进入系统钥匙串或服务器密钥服务，不进入 Renderer、`localStorage`、URL、日志、Prompt、Skill 或项目目录。
- 登出、停用、离职、团队变更和项目撤权清空可恢复令牌密文并使所有入口收敛失效。
- 管理员敏感操作要求新鲜登录；客户端把 `fresh_auth_required` 等内部错误翻译为可执行中文提示。

## 项目授权与人工确认

- Den 组织/团队只决定公司身份和能力目录，不直接决定 Brand OS 项目审批权。
- Brand OS 每次请求校验员工、项目、动作和保密级别；缓存授权必须带版本和短过期。
- PostgreSQL 运行角色不拥有表、没有 `BYPASSRLS`；事务只用 `SET LOCAL` 注入上下文。
- MCP、Agent、Workflow 和 System Actor 使用独立服务动作白名单，永远没有 `PROPOSAL_REVIEW` 或 `ACCESS_MANAGE`。
- Tool Permission 与业务确认使用不同按钮、文案、Schema、路由和审计。
- 正式写入需要幂等键和预期版本；冲突必须显示差异并重新确认。

## 信息类型与模式防护

`VIEW`、`PREFERENCE`、`HYPOTHESIS`、`OPTION`、`TENDENCY`、`TARGET_DATE` 和 `OPEN` 不能自动升级为 `DECISION`、`CONSTRAINT` 或外部 Deadline。明确决定人、决定动词、范围和证据只能形成候选，最终仍需有权限员工确认。

探索、评估、决策和执行是不同模式。AI 可以建议切换，不能自行切换。执行模式不能重开已批准战略，探索模式不能强行制造唯一答案。

## 原件与上传安全

- FoxWork 只上传员工选定且项目允许的文件；不扫描个人目录或 NAS。
- 服务端流式校验大小、MIME、SHA-256、分片校验和与安全策略，拒绝压缩炸弹、路径穿越和异常容器。
- 文件先进入隔离区，只有 `ACTIVE` 版本可以成为证据。
- 同名异内容保留新版本；删除先产生获授权撤销事件和延迟墓碑。
- 解析器在隔离 Worker 中运行，限制 CPU、内存、时长、网络和临时磁盘；失败不改变原件。
- OCR、转写和分析结果带处理器版本与原件定位，不能伪装原始内容。

## MCP、Skills 与模型

- Brand OS MCP 在 Den 目录中按成员或团队授权，但每次执行仍校验 OAuth、项目 Scope 和工具白名单。
- MCP 输入 Schema 拒绝额外字段；项目 ID 由令牌/服务器上下文固定，不能由模型切换。
- 撤权必须覆盖目录、搜索、连接和执行，不允许旧缓存继续调用。
- Skill 有版本、签名、兼容范围和回滚，不保存项目实时事实、原文、员工令牌或模型密钥。
- 共享模型凭据只由 Den 保存和注入；Brand OS 只记录模型 ID、运行版本、Task Packet 哈希和结果引用。
- 模型切换复用同一 Task Packet；多数模型一致不能替代人工批准。

## Dify 与外发

自托管不等于不外发。以下都视为离开公司控制边界：

- Dify 调用外部模型；
- 插件、HTTP 节点、网页抓取、外部转录或 Embedding；
- Open Notebook、Nubase、FlowLong 连接外部服务；
- Den Provider 指向第三方模型 API。

每次外发记录项目、任务、提供商、数据级别、证据范围、用途、保留策略和授权。P2/P3 原文默认禁止外发，除非策略和具名员工授权同时允许。外部组件输出只形成 Artifact/Proposal，并通过 Schema、证据和幂等检查回到 Brand OS。

## 内网 HTTP 安全边界

FoxWork 可以连接公司明确配置的内网 HTTP 入口。该能力只解决“公司内网不部署 TLS”这一部署选择，不降低应用层安全：

- 入口限私网/VPN/覆盖网络，禁止暴露公网；
- 不提供“忽略所有证书错误”或“允许任意来源”开关；
- PKCE、state、nonce、Origin/回调校验、短期令牌和撤权继续启用；
- Token 不进入 URL、浏览器历史或明文日志；
- 管理员和高敏操作优先要求 MFA/新鲜登录；
- F3.3/F4.7 必须分别测试重放、来源伪造、中间人风险说明和错误配置阻断。

一旦入口跨越不可信网络，必须改用 HTTPS；不能把内网例外扩展为普遍生产规则。

## Electron 与本机边界

- 主窗口启用 Sandbox 和 `contextIsolation`，preload 只暴露稳定、窄化、可校验的方法。
- IPC、深链、导航、下载、外部协议、更新源、MCP 地址和 Den 入口使用允许列表。
- 路径访问复核真实路径、符号链接、工作区边界和公司策略。
- 终端命令显示作用目录和权限；“始终允许”只属于本机工具策略，不进入业务批准。
- 服务器和 Den Worker 不能主动挂载或浏览员工电脑；依赖本机的任务在设备离线时暂停。
- 无公司配置时不得回退 PostHog、上游 Den/Cloud、公共模型目录、更新源或未登记外联。

## 供应链与许可

- OpenWork 基线固定 `v0.17.36@ddf3e482`；生产构建固定公司补丁 SHA 和镜像摘要。
- `ee/**` 按 `FSL-1.1-MIT` 管理，只用于公司内部，不包装为对外竞争服务。
- 内部分发保留 MIT/FSL/第三方许可证、版权和 SBOM，不使用 OpenWork 商标暗示官方关系。
- 依赖、容器、桌面产物、更新清单和安装包做漏洞、秘密、签名和来源扫描。
- FoxWork 更新由公司签名并可回滚；服务端保持至少一个客户端发布周期兼容。

## 日志与审计

- 业务审计记录主体、项目、动作、旧新值、证据、状态版本、幂等键和请求关联。
- Den 审计记录账号、组织、能力授权、管理员新鲜登录和撤权；Brand OS 不伪造这些事件。
- 日志不记录 Token、密码、模型 API Key、完整原文、未脱敏 Prompt、对象签名 URL 或系统钥匙串内容。
- MCP/Agent 调用记录工具 ID、参数摘要、项目、令牌主体、结果状态和 Proposal/Artifact 引用。
- 安全告警覆盖跨项目拒绝、撤权后调用、重放、异常上传、秘密扫描、签名失败和备份失效。

## 阶段门

### F3.3 Den 部署门

- 固定版本、配置、密钥、迁移、健康、日志和备份恢复可重复；
- MySQL 恢复能核对唯一组织、账号、能力授权和撤权；
- 远程工作区/Worker 可创建、分配、停止、恢复、撤权和清理，且不保存正式状态；
- 内网 HTTP 与 HTTPS 模式边界明确，不靠全局不安全开关。

### F3.4-F3.6 身份与客户端门

- 自助注册、第二组织拒绝、一次登录、无第二套账号、员工端/管理员后台全量中文和桌面交接通过；
- Brand OS audience、组织声明、项目映射和撤权联动通过；
- Den 管理员不能自动获得业务审批权。

### F3.7-F3.13 业务与能力门

- 多媒体原件、处理状态、Artifact、证据、Proposal 和人工确认闭环通过；
- 本机桥接不可被服务器绕过；MCP/Skills/模型按成员/团队下发并可撤销。

### F3.19 联网产品门

- 单账号、全中文管理面、远程工作区、多媒体、AI 能力和人工确认端到端通过；
- Den 远程工作区/Worker 必须真实通过；核心业务在 Dify、Zvec、Open Notebook、Nubase 和 FlowLong 关闭时仍通过；
- 未通过前不接真实鸿喜达资料、不向团队分发。

### F4.10 生产门

- 真实成员、并发、故障、恢复、外发、容量、签名更新和回滚均有证据；
- 业务一票否决、双主、静默覆盖、跨项目读取和撤权残留均为 0；
- Fox 明确批准范围、SLO、责任人和已知限制，否则延长试点或 No-Go。
