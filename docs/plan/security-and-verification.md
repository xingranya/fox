# 安全与验收计划

## 当前适用范围

本计划约束从 Phase 1 本地纵切到 Phase 4 团队生产准入的完整路线。团队服务器已经批准，但必须在唯一客户端纵切通过后分阶段实施。

当前验证顺序是：

1. 先证明系统正确理解品牌项目，并完成唯一客户端本地纵切。
2. 再建立服务器权威、身份权限、一致性、审计和恢复。
3. 再接入远程 MCP、Skills、Dify 和外部组件，每项保留 NoOp 退出。
4. 最后用真实团队工作验证安全、稳定和提效，决定生产范围。

本计划与[Phase 0 黄金测试集](phase-0-golden-test-set.md)、[运行时品牌 Agent 与会议解释协议](runtime-brand-agent-and-meeting-protocol.md)、[架构与接口契约](architecture-and-contracts.md)共同构成全部阶段的准入门。

## 当前安全目标

1. AI 只能读取本轮任务所需的最小、当前有效上下文。
2. 原始资料不可被模型摘要、索引或状态投影覆盖。
3. 任何模型、Skill、MCP、CLI、OpenWork/OpenCode 会话或外部工作流都不能批准业务状态。
4. 新资料只产生增量 Proposal；未确认内容不能静默进入当前状态。
5. 探索与执行模式显式隔离，防止创意开放性被工程式自动收口破坏。
6. 重要结论必须回到原始文件、会议、发言人、时间和适用范围。
7. 本机工作区、服务器凭据、客户机密和模型外发路径可枚举、最小化并可撤销。
8. 客户端缓存、索引、模型摘要和第三方数据可删除、可重建，不影响权威状态。
9. 团队阶段只有 PostgreSQL/S3 权威层可写正式状态；本地库不得与服务器形成双主。

## 当前信任边界

| 区域 | 允许 | 禁止 | 主要控制 |
|:---|:---|:---|:---|
| 鸿日原始资料区 | Fox 明确纳入的只读原件 | AI 原地改写、自动删除、扫描工作区外目录 | 路径允许列表、SHA-256、只读打开、版本登记 |
| 权威状态区 | Phase 1 本地事件；切换后 PostgreSQL 事件、审批和投影 | 模型直接写表、覆盖历史、无版本写入、长期双写 | 应用服务、幂等键、预期版本、追加事件、审计、恢复 |
| 探索与 Proposal 区 | VIEW、偏好、假设、选项、倾向、草稿 | 自动升级为决定、约束、外部承诺 | 独立 Schema、状态机、人工确认队列 |
| 本地派生区 | FTS/向量索引、摘要、Task Packet 缓存 | 成为当前有效性或批准的唯一依据 | 世代号、水位、可重建、回权威层复核 |
| 唯一员工客户端 | 查看、回源、比较、确认、驳回、本机工具 | 绕过应用服务直写底层存储或把 Tool Permission 当批准 | OIDC、窄化 IPC/API、Schema、路径复核、确认动作分离 |
| 服务器 API/MCP | 按身份和项目 Scope 读取、创建 Proposal、执行人工路由 | 服务账号批准、跨项目读取、任意 SQL/文件 | OIDC/OAuth、RBAC/RLS、工具白名单、路由隔离和审计 |
| Agent Runtime | 读取 Task Packet、使用获准工具、生成产物/Proposal | 业务批准、永久凭据、跨项目目录访问 | 任务级 Scope、目录/网络允许列表、Tool Permission 隔离 |
| 外部模型/服务 | 经授权的最小脱敏 Task Packet | 默认接收 P2/P3 原文、密钥、完整项目镜像 | 数据分级、提供商登记、任务授权、留痕和退出 |

OpenWork Server、OpenCode、Dify、FlowLong、Open Notebook、Nubase 和 Zvec 即使在本机运行，也属于非权威组件。

## 信息分类与状态升级防护

### 必须区分的对象

| 类型 | 含义 | 默认状态 | 能否直接改变当前状态 |
|:---|:---|:---|:---:|
| `FACT` | 有来源的客观陈述候选 | proposed | 否 |
| `VIEW` | 某人的看法 | working | 否 |
| `PREFERENCE` | 审美或方式偏好 | working | 否 |
| `HYPOTHESIS` | 待验证解释 | working | 否 |
| `OPTION` | 可继续探索的选择 | working | 否 |
| `TENDENCY` | 当前倾向而非承诺 | preferred | 否 |
| `TARGET_DATE` | 目标、评审或暂定时间 | tentative | 否 |
| `DECISION_CANDIDATE` | 具备部分决定信号的候选 | proposed | 否 |
| `DECISION` | 具名人员确认的正式决定 | approved | 仅人工批准事件 |
| `CONSTRAINT` | 有范围和来源的正式约束 | approved | 仅人工批准事件 |
| `ACTION` | 具备负责人、动作和时间性质的行动候选 | proposed | 否 |
| `OPEN` | 仍待回答的问题 | open | 否 |

明确决定人、决定动词、适用范围和证据只能让系统提出 `DECISION_CANDIDATE`；最终升级仍需 Fox 或未来获授权人员明确确认。

### 日期性质

- `EXTERNAL_DEADLINE`：外部正式承诺或法律/合同截止，必须有来源和适用范围。
- `INTERNAL_TARGET`：内部希望完成时间。
- `REVIEW_POINT`：评审或检查节点。
- `TENTATIVE_DATE`：暂定、最好、争取等表达。

非 `EXTERNAL_DEADLINE` 不得在界面、任务包或模型输出中显示为“死线”“必须交付”或同义表达。

## 工作模式安全

| 模式 | 允许 | 禁止 |
|:---|:---|:---|
| 探索 | 保留矛盾、提出假设、发展不同战略领地、说明取舍 | 自行锁定唯一方向、直接写正式状态、用完整交付掩盖策略缺口 |
| 评估 | 对照证据和标准比较选择、指出未知项 | 伪造数据、把排序当作批准 |
| 决策 | 装配候选、证据、影响和待确认问题 | 由 AI 代替人批准、缺证据时强行收口 |
| 执行 | 严格遵守批准方向、事实、格式和验收 | 重开战略、带回废案、擅自改变约束 |

模式切换必须由具名人类动作或明确的已批准任务指令触发，并记录前后模式、操作者、时间和状态版本。模型不得根据“信息看起来够了”自动切换。

## Task Packet 与多模型一致性

每个模型领取同一结构和版本的 Task Packet：

1. 项目与当前状态版本。
2. 当前阶段、本轮任务、目标和非目标。
3. 已批准决定与约束。
4. 开放问题和待确认冲突。
5. 本任务相关证据摘要及回源句柄。
6. 当前工作模式、运行时品牌角色和质量标准。
7. 输出 Schema、允许动作、禁止动作和数据外发边界。

Task Packet 不包含无关全量历史、默认废案、永久凭据或模型私有聊天记忆。Codex、Claude 或其他模型的表达可以不同，但不得获得互相冲突的正式事实。

## 本地数据与凭据

- 默认只读取 Fox 明确选择的项目根目录；禁止隐式扫描 Home、Downloads、云盘或其他客户目录。
- 原件入册记录路径、SHA-256、大小、MIME、版本和数据级别；内容变化形成新版本，不原地伪装成同一原件。
- 本地数据库、事件日志和配置使用独立应用目录；不得散落到项目原件目录或代码仓库。
- 敏感配置只进入系统钥匙串或环境注入；不得进入 Renderer、`localStorage`、URL、日志、Prompt、Git 或 Skill。
- P2/P3 原文默认不得发给外部模型、Embedding、STT/TTS、插件、HTTP 节点或遥测服务。
- 本地备份必须包含权威事件、状态版本、审批与原件清单；索引和摘要可重建，不作为备份完成依据。
- 恢复后运行黄金测试的状态、回源和权限烟测；恢复失败不得继续试点写入。

## 桌面、MCP、CLI、Skills 与 Agent Runtime

### 唯一员工客户端

- 客户端只通过版本化应用端口读取状态、回源和提交 Proposal，不直接写底层数据库。
- 客户端基于经许可审查的 OpenWork MIT 社区核心；OpenWork Server/OpenCode 只作 Agent 运行层，不是 Brand Project OS Service。
- Electron 必须启用 Sandbox、`contextIsolation`、最小 preload 和 IPC 允许列表；主进程复核路径、Scope 与 Schema。
- Tool Permission 只表示本次运行可以读文件、执行命令或调用工具，不能生成业务批准事件。

### MCP 与 CLI

- 首版允许：读取当前状态/Task Packet、检索和回源证据、查看决定/开放问题、创建 Proposal、运行诊断。
- 首版禁止：批准/驳回、任意 SQL、原件硬删除、密钥读取、工作区外文件访问和第三方管理面透传。
- 工具参数与结果必须通过 JSON Schema；每次调用绑定项目、状态版本、任务和最小 Scope。
- Phase 1 stdio MCP 使用本地项目固定 Scope；Phase 3 远程 MCP 使用短期 OAuth/服务令牌和项目 Scope。两者都不能调用人工批准。Codex/Claude 的模型提供商登录由各自运行时管理，Brand Project OS 不读取或转存 API Key。
- MCP 项目在进程启动时固定，输入 Schema 拒绝额外字段；默认 10 秒、最高 60 秒超时，客户端取消必须传播。写请求超时后只能使用同一幂等键查询或重试。

### Skills

- Skills 只保存可重复流程、工具顺序、Schema 和失败处理。
- Skills 不保存鸿日事实、聊天摘要、原始客户资料、密钥或数据库连接串。
- Skill 版本变化不得改变当前状态；运行结果仍只能形成 Proposal。

## 必测威胁

| 威胁 | 防护 | 必测结果 |
|:---|:---|:---|
| 讨论被升级为决定 | 独立分类、迁移规则、人工批准 | G-03 至 G-06 无非法升级 |
| 暂定日期变成死线 | 日期性质 Schema 与显示规则 | G-05 保持 `tentative` |
| 旧方案污染当前方向 | 有效期、`supersedes`、固定读取顺序 | G-02 不把废案带入当前答案 |
| 模型虚构事实 | 最小 Task Packet、证据要求、未知显式化 | G-14 明确未确认且不补全 |
| 新会议覆盖历史 | 增量事件、Proposal、版本比较 | G-08/G-09 只产生变化和冲突 |
| 提示注入或伪 `FINAL` | 原文视为数据、AI 无批准能力、工具允许列表 | G-15 正式状态不变 |
| Tool Permission 冒充业务批准 | 不同 Schema、路由和审计 | 允许工具后状态版本不变化 |
| 跨目录读取 | 工作区根允许列表、真实路径复核、符号链接检查 | `..`、符号链接和未选目录均拒绝 |
| 本地数据库损坏 | 追加事件、校验、原子备份和恢复演练 | 恢复后状态与证据链一致 |
| 外部模型泄密 | 分级、授权、最小披露、提供商登记 | P2/P3 未授权外发为 0 |
| 模型切换产生事实分叉 | 相同 Task Packet 版本与契约 | G-13 正式事实和证据一致 |
| 派生组件污染状态 | 单向输入、Schema、Proposal 门、可删除重建 | 删除索引/摘要后权威状态完整 |

## 测试分层

| 层级 | 当前核心用例 | 通过标准 |
|:---|:---|:---|
| 领域单元 | 分类、日期性质、状态迁移、替代、冲突、模式切换 | 所有非法迁移拒绝，AI 无批准路径 |
| Schema/契约 | 项目状态、事件、证据、Proposal、Task Packet、MCP、错误码 | 所有入口共享版本化契约 |
| 本地持久化 | 事务、幂等、预期版本、备份、恢复、索引重建 | 无部分提交、静默覆盖或恢复后漂移 |
| 摄取/解析 | 哈希、版本、重复文件、会议分段、原话定位、不可信内容 | 原件不可被摘要替代，重复摄取不重复改变状态 |
| 检索与回源 | 当前有效性、权限、关系遍历、原文定位 | 已确认关键结论 100% 回源 |
| MCP/CLI/Skills | 工具表、Scope、Schema、钥匙串、禁止动作 | 多入口状态版本一致，无事实或密钥进入 Skill |
| 客户端 | 当前状态、增量差异、证据、确认、模式、空错态、键盘 | 关键旅程可完成，无绕过服务直写 |
| Agent Runtime | 任务身份、Tool Permission、取消、产物、运行时替换 | 权限不变成批准，停用运行时后状态仍可读 |
| 黄金 E2E | G-01 至 G-16 | 硬门全部通过，一票否决为 0 |
| 品牌人评 | 匿名、随机顺序、普通聊天基线 | 达到 Phase 0 冻结的质量准入线 |
| 真实使用 | 解释耗时、纠错、返工、会议确认、可用性 | Phase 2 有可复核改善证据 |

## 发布门

### Phase 0 完成门

- Fox 确认两条需求线拆分和一页 MVP 边界。
- 10-20 个黄金用例、Fixture Manifest、分类标准和评分表冻结。
- 运行时品牌 Agent、探索协议、执行规格和模式切换规则明确。
- 一票否决及人工批准边界进入机器可测试 Schema。

### Phase 1 单一客户端门

- 冷启动、会议解释、回源、增量更新和模型切换的确定性硬门通过。
- 原始资料只读、事件/Proposal 可追溯、本地备份可恢复。
- Codex 与 Claude 可通过同一版本入口读取 Task Packet。
- 公司定制 OpenWork 可以查看状态、证据和待确认变化；AI 无批准控件；不需要第二个客户端。

### Phase 2 服务器基础门

- PostgreSQL/S3、OIDC/RBAC/RLS、幂等/乐观锁、审计/Outbox、API 和恢复均通过。
- 服务账号、MCP、Skill 和工作流无法批准正式状态。
- 跨项目、撤权、并发冲突、日志脱敏和恢复对账有自动化与演练证据。

### Phase 3 联网与集成门

- SQLite 已完成一次性迁移并退出正式写入。
- 远程 MCP、Skills、Dify 和每个外部组件分别通过许可、外发、权限、故障和 NoOp 验收。
- 服务器不能绕过客户端访问本机文件、终端和桌面。

### Phase 4 团队生产门

- 真实成员、并发、故障、恢复、撤权、外发、容量、签名更新和回滚均有证据。
- 真实工作和 BrandBench 达到基线，七项业务一票否决、双主和静默覆盖均为 0。
- Fox 明确批准生产范围、责任人、内部 SLO 和已知限制。

```bash
uv run pytest
uv run brand-os doctor
uv run brand-os verify hongri
```

服务器测试、OpenWork 客户端类型检查、桌面 E2E、产物扫描和恢复演练命令在各任务实现时冻结。未经用户明确要求，不启动 Web、数据库、Docker、桌面应用或其他常驻服务；fraimz 验收可短时启动桌面应用并在完成后立即退出。
