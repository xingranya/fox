# 任务分解

## 总览

- 当前目标：交付基于 OpenWork 的唯一员工客户端，以及部署在公司服务器上的 Brand Project OS Service。
- 第一用户：Fox；第一验证项目：鸿日。
- 员工只安装一个客户端。OpenCode、Sidecar 和本机桥接随包交付；不再开发第二个 Web、PWA 或桌面客户端。
- 团队正式状态由服务器权威服务保存。MCP 是 AI 接口，不是数据库或审批核心。
- Phase 0 和 F1.1-F1.8 的成果继续有效；原“先真实试用、再决定服务器”的 Phase 2/3 尚未开始，现由新的 Phase 2-4 取代。
- 追踪模式：`LOCAL_ONLY`。共 5 个阶段、49 项任务，当前完成 21 项。

## 产品与数据边界

```text
员工 -> Brand Project OS Desktop（公司定制 OpenWork，唯一客户端）
     -> Brand Project OS API -> 领域应用服务 -> PostgreSQL / 对象存储

Codex / Claude / Dify / 其他 Agent
     -> MCP Gateway / Skills -> 同一领域应用服务

客户端本机桥接 -> 只处理用户授权的文件、终端和桌面动作
```

- Phase 1 结束前，本地 SQLite 仍是验证环境权威。
- Phase 3 切换完成后，PostgreSQL 是唯一可写正式状态源；对象存储保存原件版本和 SHA-256。
- 本地 SQLite、OpenWork/OpenCode Session、缓存、索引、Dify、Zvec、Open Notebook、Nubase 和 FlowLong 都不能成为正式状态源。
- AI、MCP、Skills、工作流和服务账号只能读取或创建 Proposal。人工确认必须来自有权限员工的交互式客户端。
- 离线状态只允许查看已标记水位的缓存和编辑草稿。恢复联网后重新校验 `expected_version`，不能离线提交正式决定。

## S.U.P.E.R、测试与提交规则

- S：领域、客户端、运行时、MCP、外部组件和基础设施各守一层职责。
- U：正式写入只走应用服务，事件、投影、审批、审计和 Outbox 保持单向事务流。
- P：所有跨层调用使用版本化 Schema、OpenAPI、MCP Tool Schema 或端口。
- E：本地开发、公司测试和生产环境只通过配置变化，不把环境地址写死在领域代码中。
- R：模型、检索、内容处理、记忆和工作流适配器都可禁用、替换并回退到 NoOp。

每个功能单独实现、测试、记录遥测并提交。任何用户行为、Schema、迁移、解析、权限、检索、缓存或持久化变化都要增加自动化测试。阶段门未通过时，不进入下一阶段。

- 启动条件：Phase 4 真实团队试点形成结论后，Fox 单独批准候选 POC；必须先完成一次正式 rescope，不能因为文档已存在就视为已批准。
- 候选任务：BS0 固定版本与安全边界、BS1 适配器与模拟测试、BS2 隔离服务器 POC、BS3 OpenWork 展示与 Go/No-Go。合法结果包括采用、延长评估或拒绝。
- 硬边界：不直连 BISHENG 数据库，不迁移正式批准，不把 BISHENG 前端嵌成第二客户端；BISHENG 停机时 Brand Project OS 核心旅程仍必须可用。

## 一票否决

1. 虚构产品事实或证据。
2. 把讨论、偏好、假设或选项升级为决定/约束。
3. 把暂定日期写成外部死线。
4. 把过期方案当成当前方向。
5. 重要结论无法回到原始证据。
6. 未经有权限员工确认改变正式状态。
7. 在探索模式下强行制造唯一答案。
8. 出现两个可写正式状态源或静默覆盖并发写入。
