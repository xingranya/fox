# 任务分解

## 总览

- 总阶段：6。
- 总任务：42。
- 团队服务器试运行版：Phase 1-4，共 29 项；三条主 Lane 并行预计 5-7 周，单人顺序实施预计 12-16 周。
- 外部组件验证与生产准入：Phase 5-6，共 13 项；工期取决于许可、真实数据和恢复演练结果，不承诺与试运行版同时完成。
- 总体工作量：XL。
- 追踪模式：`LOCAL_ONLY`。
- 生产权威：PostgreSQL 保存正式状态、领域事件、审批、权限、审计与 Outbox；S3 兼容对象存储保存按 SHA-256 寻址的不可变原件。
- SQLite 边界：只用于开发、自动化测试、单机演示和服务器导出的只读应急快照，不接受团队生产写入，也不得与服务器形成双主。

详细架构与操作约束见[团队服务器架构](team-server-architecture.md)、[数据一致性与可靠性计划](data-consistency-and-reliability.md)和[前端与 AI 访问规划](frontend-and-ai-access.md)。

## 全局 S.U.P.E.R 约束

- **S - 单一职责**：领域、身份、对象存储、导入、检索、审批、AI 工作流、外部适配器、运维和 UI 分层实现。
- **U - 单向数据流**：数据只按“原件准入 -> 候选 -> 人工批准 -> 权威事件/投影 -> Outbox -> 派生层”流动；外部回调不能反写正式状态。
- **P - 端口优先**：先冻结版本化 Schema、端口、错误模型、幂等键和并发语义，再实现 PostgreSQL、S3、MCP、Dify 等适配器。
- **E - 环境无关**：域名、路径、端口、身份提供商、密钥、模型和组件实现均由配置注入；生产密钥不得进入仓库或浏览器。
- **R - 可替换**：PostgreSQL FTS/Zvec、直接 Worker/Dify、核心审批/FlowLong、内置解析/Open Notebook、独立身份/Nubase 能力均通过端口和 NoOp 适配器切换。

每项任务的验收均隐含：通过所列原则的 S.U.P.E.R Quick Check，且领域核心不直接导入 PostgreSQL、S3、Zvec、Open Notebook、Nubase、FlowLong、Dify 或具体模型 SDK。

## 测试与治理约束

- 新增或修改 Schema、迁移、身份、RLS、解析、检索、审批、API、MCP、缓存、工作流或持久化逻辑，必须增加对应自动化测试。
- 纯文档或配置任务必须说明不增加行为测试的理由，并运行链接、Schema、配置或静态校验。
- 所有正式写接口必须带 `idempotency_key` 和 `expected_version`；冲突返回可处理的 `409`，不得静默覆盖。
- 业务写入、领域事件、最小正式投影、审计关联信息和 Outbox 必须在同一 PostgreSQL 事务提交。
- Web/PWA、远程 MCP、CLI、Skills、Dify 和其他自动化只调用统一应用服务，不持有生产数据库凭据。
- AI、Skill、MCP、Dify、Open Notebook、Nubase Memory 和 FlowLong 均无批准权限，只能读取最小上下文或创建 Proposal。
- 业务事实不进入 `AGENTS.md`、`CLAUDE.md`、Skill 或对话记忆；稳定工程不变量更新共享指令面和 ADR。未经用户批准不创建仓库内 Memory。
- 每项任务完成后先记录实际工时、S.U.P.E.R 分数、未计划依赖和漂移，再更新阶段文件与 `docs/progress/MASTER.md`。

## 全局停止条件

- 出现一次影响正式状态或对客内容的未经授权升级。
- 任一已批准事实、决定、约束、负责人、截止时间或提交版本无法完整回源。
- 同一业务状态由两个数据库、离线客户端或外部工作流同时接受正式写入。
- 跨工作空间或跨项目读取、对象访问、缓存或 AI 上下文泄漏。
- PITR、对象恢复或权威事件重放无法在目标时间内通过验证。
- Dify 源码多租户/白标、FlowLong 嵌入/分发或任何真实数据 POC 未先通过对应书面许可与隐私门。
- 单场会议人工审批超过 10 分钟、默认高价值卡超过 8 张或待确认积压持续超过 20 项；先优化候选聚类和风险分级，不以自动批准消化队列。
