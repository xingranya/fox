# 任务分解

## 总览

- 总阶段：4，编号为 Phase 0-3。
- 总任务：32。
- 当前验证主线：Fox 单用户、本地优先、鸿日真实项目、先证明理解正确和真实提效。
- 当前实现边界：本地只读原件、SQLite/FTS 等轻量结构化存储、本地 API/CLI/MCP、Codex/Claude 统一 Task Packet、简单查看与确认界面。
- 后续决策边界：团队服务器、PostgreSQL、S3、OIDC、RLS、并发、高可用、灾备、OpenWork 深度团队化及五个外部组件只在 Phase 3 评估，不是 Phase 1 前置。
- 追踪模式：`LOCAL_ONLY`。
- 预计工作量：XL；Phase 2 必须跨真实工作自然发生，不能用一次演示代替连续验证。
- 需求基线：`20260713_品牌AI长期项目协作系统_场景化需求澄清与产品反馈_v0.1(1).md`。

本计划替代此前 6 阶段 42 项、全部未开始的“团队服务器先行”方案。旧任务的逐项去向见文末[旧 42 项追踪表](#旧-42-项追踪表)。替代不表示删除未来架构，而是把未经鸿日验证的服务器与组件设计降级为 `future-candidate`。

## 产品验证顺序

```text
先定义什么叫理解正确
-> 做出 Fox 可用的鸿日本地原型
-> 在真实工作中连续验证理解、证据和品牌质量
-> 只有价值门通过，才决定是否以及如何团队服务器化
```

两条需求线必须分开：

| 需求线 | 当前地位 | 本计划处理方式 |
|:---|:---|:---|
| 品牌 AI 长期项目协作 | 当前 P0 主线 | Phase 0-2 以 Fox 和鸿日完成价值验证 |
| FoxWork 团队文件协作 | 独立后续议题 | 仅在 F3.1 判断是否合线，不预设共享服务器方案 |

## 全局 S.U.P.E.R 约束

- **S - 单一职责**：原件准入、内容索引、会议解释、当前状态、证据关系、Proposal、Task Packet、模型适配、界面和评测分开实现。
- **U - 单向数据流**：只允许“只读原件 -> 解析/分类候选 -> Fox 确认 -> 当前状态/历史 -> Task Packet/输出”；AI 输出不能反向覆盖原件或已确认状态。
- **P - 端口优先**：先冻结可序列化 Schema 与 Port，再实现 SQLite、本地文件、CLI/MCP、模型和界面；未来换 PostgreSQL/S3 或客户端不得改领域语义。
- **E - 环境无关**：`/Users/fox/work` 是第一使用场景，不得硬编码；路径、模型、密钥、缓存和测试夹具由配置注入，测试使用临时目录。
- **R - 可替换**：SQLite/未来服务器、内置 FTS/未来索引、Codex/Claude/其他模型、简单界面/未来桌面客户端都通过端口替换。

每项任务验收均隐含通过所列原则的 S.U.P.E.R Quick Check。领域核心不得直接依赖 OpenWork、OpenCode、Dify、FlowLong、Nubase、Open Notebook、Zvec 或具体模型 SDK。

## 测试与治理约束

- Phase 0 先建立 10-20 个黄金用例和品牌质量基线；实现任务不得以“尚无测试”为由后移这些门。
- 新增或修改 Schema、迁移、解析、分类、状态机、检索、缓存、持久化、API、MCP、CLI、界面或模型适配必须增加自动化测试。
- 人工匿名评审是品牌质量验收的一部分，不能用接口成功率、代码覆盖率或模型自评代替。
- 原始资料只读并记录来源、版本和哈希；AI 生成物、摘要、索引和聊天记忆均为派生数据。
- AI 只能创建 Proposal。任何事实、决定、约束、负责人、截止时间、提交版本和工作模式切换都由 Fox 显式确认。
- 开发仓库指令面只保存工程规则；品牌 Agent 宪法、工作模式协议和鸿日项目规则分层维护，业务事实不写入 `AGENTS.md` 或 Skill。
- 当前未获准建立仓库 Memory；稳定工程规则更新既有指令面/ADR，业务事实只进入本地权威状态与证据链。
- PostgreSQL、S3、OIDC、RLS、HA/灾备、团队账号和五个外部组件在 Phase 3 之前不得成为实现依赖或验收门。
- 每项任务完成前先记录实际工作量、S.U.P.E.R 分数、未计划依赖和任务漂移，再更新阶段文件与 `docs/progress/MASTER.md`。

## 一票否决

以下任一情况出现一次，本次用例、模型/协议版本和所属阶段即失败，不能用平均分、功能完成度或人工补救掩盖：

1. 虚构产品事实。
2. 把讨论升级成决定。
3. 把暂定日期写成死线。
4. 把过期方案当成当前方向。
5. 重要结论不能回到证据。
6. 未经人确认自动改变项目状态。
7. 在探索模式下强行制造唯一答案。

失败后必须保留输入、输出、模型/协议版本、状态版本和证据水位，新增或修正 Fixture，完成修复和全量回归后才能重新过门。
