## M0：理解正确性基线

**完成条件**：已通过。

- 鸿日来源、版本、SHA-256、授权和缺口可复核。
- 讨论、偏好、假设、选项、倾向和暂定日期不会自动升级。
- 探索与执行模式由 Fox 显式切换。
- 黄金集覆盖八条旅程和七项一票否决。
- BrandBench 以匿名人工评分为准，已记录首轮失败和重写后的通过基线。

## M1：单一客户端本地纵切

**目标**：员工只安装并打开公司定制 OpenWork，就能完成品牌项目的核心工作。

### 完成条件

- F1.1-F1.8 的本地领域、证据、Task Packet、CLI 和 stdio MCP 回归保持通过。
- OpenWork 固定在审查过的稳定 SHA；默认无上游遥测、Cloud、模型目录、更新和宽松网络出口。
- OpenCode Runtime、Sidecar 和本机桥接随同一安装包交付，运行健康可诊断。
- 首屏可读当前阶段、任务、决定、开放问题和最近变化；可回源、运行 AI、查看 Artifact 和处理 Proposal。
- Tool Permission 与业务确认使用不同路由、Schema、权限和审计。
- fraimz 桌面旅程、黄金集、空错加载、安装烟测和真实鸿日纵切通过。
- 七项一票否决为 0；删除 OpenWork/OpenCode 运行数据不损坏正式状态。

### 阻断条件

- 真实资料接入前仍存在未登记的上游外联。
- 员工还需安装第二个主客户端或手工管理 OpenCode/Sidecar。
- 会话、工具允许或本地缓存可以绕过人工确认改变正式状态。
- 只有打包和测试数字，没有实际桌面旅程证据。

## M2：服务器权威基础

**目标**：把团队正式状态放到公司服务器，并建立可恢复、可审计、无双主的写入体系。

### 完成条件

- PostgreSQL 保存权威事件、审批、当前投影和审计；对象存储保存原件版本和 SHA-256。
- OIDC 员工身份、项目 RBAC、保密级别和 RLS 防线通过正反权限测试。
- 每个写请求有幂等键和 `expected_version`；并发冲突返回 409 和差异，不做最后写入覆盖。
- 事件、审批、投影、审计和 Outbox 同事务提交；消费者幂等并可重放。
- Desktop API 与 Agent API 分路。服务账号、MCP、Skill 和工作流没有人工批准权限。
- 日志、指标、追踪、健康和告警能定位请求、项目和状态水位，且不泄露原文和密钥。
- PostgreSQL、对象存储和投影完成恢复演练，恢复后事件、审批、投影和哈希对账一致。
- Fox 确认内部 SLO、RPO、RTO 和当前部署档位；这些是内部运行目标，不是外部承诺。

### 阻断条件

- 客户端直连数据库/对象存储，或 OpenWork Server/OpenCode Session 成为业务服务。
- 出现第二个可写权威源、静默覆盖或无法重放的状态。
- 服务账号可以调用人工确认处理器。
- 只有备份文件，没有实际恢复和对账记录。

## M3：联网产品与 Agent/工作流接入

**目标**：唯一客户端切到服务器权威层，其他 Agent 通过 MCP/Skills 使用同一项目认知。

### 完成条件

- SQLite 数据一次性迁移并完成 ID、事件、审批、投影、来源和 SHA-256 对账；切换后本地库只读。
- OpenWork 内完成登录、项目选择、状态/证据、Proposal、冲突处理和本机工具授权。
- 服务器不可达时客户端明确进入只读/草稿状态，缓存显示版本和水位。
- MCP Gateway 对 Codex/Claude 提供项目 Scope 固定的白名单工具；没有批准、任意 SQL 或任意本机文件能力。
- Skills 有版本、签名、兼容和回滚；Skill 不保存实时事实。
- Dify 通过 `AIWorkflowPort` 接入，外部模型/插件/HTTP 节点按数据外发处理。
- Zvec、Open Notebook、Nubase、FlowLong 分别完成许可、数据流、质量、故障和 NoOp 评估；逐项记录采用或拒绝。
- 不安装第二个客户端即可完成 Desktop、MCP/Skills 和工作流端到端旅程。

### 阻断条件

- 本地与服务器长期双写，或离线缓存可直接提交正式决定。
- 远程 MCP 绕过客户端读取员工电脑。
- 为接入外部组件修改领域真相语义，或组件退出后无法运行核心旅程。
- 许可未清、外发未授权或真实数据被发送到未登记处理方。

## M4：团队试点与生产准入

**目标**：在真实团队工作中证明系统稳定、可恢复、不会误解人，也确实减少重复解释。

### 完成条件

- 试点成员、角色、项目、资料和外发范围有明确授权。
- 多客户端并发、重试、断线重连和陈旧缓存测试无丢失、无静默覆盖、无双主。
- 至少两次真实会议或资料增量，以及一轮“探索 -> 人工选择 -> 执行”闭环完成。
- Codex/Claude 使用同一 Task Packet 时，事实、决定和证据一致；品牌结果达到人工基线。
- API、数据库、对象存储和队列故障均完成只读降级、恢复和对账。
- 身份撤权、跨项目隔离、审计、外发、SBOM 和依赖风险通过检查。
- 容量、性能、成本和告警来自真实试点测量；部署档位和 SLO 与测量一致。
- 客户端完成公司签名、内部更新、灰度和坏版本回滚。
- Fox 明确记录生产 Go、延长试点或 No-Go，并批准范围、责任人、运行目标和已知限制。

### 阻断条件

- 以单人演示或合成样本代替真实团队使用。
- 以平均分掩盖一票否决、安全越权或数据一致性错误。
- 只验证正常路径，没有故障、恢复、撤权和回滚证据。
- 未经 Fox 明确批准就宣称生产可用或扩大资料范围。

## M4 之后的候选门：BISHENG

BISHENG 不属于 M0-M4，也不计入当前 49 项。只有 Phase 4 形成真实团队试点结论，且 Fox 单独批准启动 POC 后，才执行 [BISHENG 条件接入 SPEC](bisheng-integration-spec.md) 的 BS0-BS3。

候选 POC 必须复用 `AIWorkflowPort`，与 Dify 或直接 Worker 使用同一输入、输出和评分口径。采用、延长评估和拒绝都是合法结果；在正式 rescope 前，不得让它成为当前里程碑的依赖。
