# 里程碑

## 排期口径

- Phase 0-1 是当前最小交付主线，共 17 项；Phase 0 先定义正确性，Phase 1 才写本地原型。
- Phase 2 共 8 项，必须跨正在发生的鸿日工作连续验证，不能压缩成一次集中演示。建议至少覆盖两次真实会议/资料增量和一轮“探索 -> Fox 选择 -> 执行”闭环。
- Phase 3 共 7 项，是决策阶段，不是团队服务器实施阶段。只有 Phase 2 的价值结论为 Go 才启动。
- 服务器、PostgreSQL/S3、OIDC/RLS、并发、HA/灾备、OpenWork 深度团队化及五组件都必须在 Phase 3 重新证明必要性；未采用也是合法结论。
- 当前没有应用代码，所有周数只用于容量预估，不构成负责人、截止时间或外部承诺。
- 追踪模式为 `LOCAL_ONLY`，总计 4 个里程碑、32 个任务。

## M0：理解正确性基线

**包含任务**：F0.1-F0.7。
**目标**：在实现前回答“当前产品验证什么、什么叫理解正确、什么情况必须失败”。

### 完成标准

- 两条需求线明确拆分：当前只验证 Fox 的品牌 AI 长期项目协作；FoxWork 团队文件协作不作为 MVP 前置。
- 鸿日第一批资料 Manifest 固定来源角色、版本、哈希、授权和已知缺口，并对账 V5 已知数量；没有真实内容时不伪造 Fixture。
- 探索/评估/决策/同步会议模式，信息类型和时间性质都有可执行真值表；决定候选的充分条件与人工最终确认明确。
- 品牌 Agent 宪法和探索/评估/决策/执行协议分离；模式切换由 Fox 显式完成。
- 10-20 个黄金用例覆盖八条真实旅程，并包含七项一票否决的正反例和可重复运行方式。
- BrandBench 完成匿名评分表、无系统辅助基线、随机化和人工裁决规则。
- 本地纵切、Schema、Port、配置和测试门冻结，明确不依赖服务器或外部组件。

### 阻断条件

- 没有冻结真实资料范围、分类真值表或 10-20 个可执行黄金用例。
- 把品牌质量简化为模型自评、接口成功或代码覆盖率。
- 开发仓库 AGENTS、运行时品牌宪法、工作模式协议和鸿日项目规则仍混为一体。
- PostgreSQL、S3、OIDC、OpenWork 深度团队化或五组件被列为 Phase 1 必需依赖。

## M1：Fox 可用的鸿日本地原型

**包含任务**：F1.1-F1.10。
**目标**：在不建设团队服务器的前提下，让 Fox 完成本地资料、会议、状态、AI 工作、人工确认和证据回查闭环。

### 完成标准

- 工作空间路径可配置；原件区只读且有哈希，派生区可重建，SQLite 状态库可迁移和备份恢复。
- 鸿日资料导入幂等，V5 旧 ID、关系、废弃保号、版本和 `KNOWN_SOURCE_GAP` 保留且数量可对账。
- 新会议只产生带原话、发言人、时间、上下文和证据的增量候选；重复去重、冲突单列、暂定日期不变 Deadline。
- 当前状态只能由 Fox 确认/修改/驳回/替代事件改变；历史可重放，决定不会被静默覆盖。
- 证据关系支持来源、提出、支持、反对、冲突、批准、替代、适用和待确认；重要结论可达原件，废案默认排除。
- Task Packet 只装配当前任务所需的状态、证据、开放项、工作模式、角色和禁区；AI 不能自行切换模式。
- Codex、Claude 通过同一本地 CLI/MCP 契约获取相同项目事实，不依赖各自聊天记忆；工具表不含批准和任意 SQL。
- 简单本地界面可看当前状态、证据、最近变化和待确认，能选择模式和发起 AI 工作；不要求 OIDC、远程服务器或完整 PWA。
- 全部黄金、备份恢复、模型切换和 E2E 通过，七项一票否决为 0，Fox 能独立开始试用。

### 阻断条件

- SQLite、模型输出、聊天摘要或界面缓存绕过人工确认成为正式状态。
- 重要结论无法回到原件/会议/发言人/时间/适用范围。
- 换模型需要 Fox 重新解释完整项目，或不同模型得到互相冲突的当前事实。
- 任何一票否决未修复，或 BrandBench 未达到 M0 预设基线。
- 为完成本地原型而启动 PostgreSQL、S3、OIDC、Dify、FlowLong、Nubase、Open Notebook 或 Zvec。

## M2：真实工作价值证明

**包含任务**：F2.1-F2.8。
**目标**：用鸿日正在发生的工作证明系统真正减少解释和混乱，并提升策略/文案质量。

### 完成标准

- 使用前价值基线可复核：重复解释次数/时间、状态错误、证据查找、会议处理、模型切换成本和品牌盲评分口径明确。
- 连续覆盖至少两次真实会议或资料增量，每次只提出变化/冲突，不静默覆盖状态，人工确认可追溯。
- 多个新会话/模型完成冷启动，准确说明阶段、目标、决定、开放问题、本轮任务和来源；随机重要结论回源完整。
- 完成至少一项真实策略探索，选择真正不同且说明获得/放弃；Fox 选择前不提前形成唯一答案。
- Fox 批准后完成至少一项命名、文案或方案落地，严格服从获批方向，不带回废案。
- Codex、Claude 至少完成一次同 Task Packet 对比，当前事实/决定/证据一致，质量、成本、时延和失败有记录。
- 真实候选完成匿名品牌评审，并与 M0 基线比较；平均分不能掩盖任何一票否决。
- 至少一个真实错误完成“复现 -> Fixture -> 规则修订 -> 全量回归”闭环。
- F2.8 给出带前后证据的 Go/延长/No-Go，并由 Fox 显式确认。

### 阻断条件

- 只演示预置样本，没有连续真实会议、策略和执行工作。
- 系统功能完整，但重复解释、状态准确、证据查找或品牌盲评没有基线/对照。
- 在探索模式强行收口，或在执行模式重新发明战略。
- 真实错误仅手工修正结果，没有进入 Fixture 与规则回归。
- 以“未来团队服务器会解决”为理由给当前价值缺口判 Go。

## M3：团队服务器化决策

**包含任务**：F3.1-F3.7。
**目标**：从 Phase 2 的真实价值和使用证据出发，决定保持本地、局部共享还是启动团队服务器化。

### 完成标准

- 明确第一批团队用户、并发协作动作、共享/私有数据和 FoxWork 是否合线；没有真实场景时保留本地。
- 对 SQLite/同步/PostgreSQL/S3 候选给出数据量、并发、迁移、冲突、成本、恢复和退出比较，不预设服务器胜出。
- OIDC/RBAC/RLS/审计/幂等/Outbox/HA/灾备只针对已确认的多人风险设计，并有触发阈值。
- OpenWork 深度团队化、远程 Agent、轻量 Web、签名更新和上游维护有收益/成本/许可/安全证据，不因旧 ADR 自动采用。
- Zvec、Open Notebook、Nubase、FlowLong、Dify 逐项根据 Phase 2 缺口形成采用/推迟/拒绝候选；许可和退出门完整，可以全部拒绝。
- 继续本地、局部共享、小团队服务器、高可用服务器等档位有成本、负责人、SLO、迁移、故障和回退说明。
- F3.7 由 Fox 批准明确 Go/No-Go；Go 时只创建新的实施 SPEC，不直接恢复旧 42 项。

### 阻断条件

- F2.8 不是 Go，或 F3.1 不能证明团队协作需求。
- 用已有技术设计、GitHub Stars 或“以后规模会大”替代真实数据、负载和用户证据。
- 未通过许可门就用真实数据验证 Dify/FlowLong，或任何组件获得正式状态写权限。
- 架构无法从本地原型迁移/回退，或需要同时维护两个可写真相源。
- 未经 Fox 明确批准就把候选架构记为已决定。

## 八条黄金旅程

| 旅程 | 里程碑通过要求 |
|:---|:---|
| 冷启动 | 新 AI 无旧聊天仍能准确说明当前阶段、目标、决定、开放问题、本轮任务和来源 |
| 会议解释 | VIEW、OPTION、HYPOTHESIS 不升级为 DECISION/CONSTRAINT，暂定目标不变 Deadline |
| 证据回查 | 返回决定人、原话、会议、时间和范围；不足时明确未确认 |
| 增量更新 | 只提出变化建议和冲突，不静默覆盖旧状态 |
| 策略探索 | 给出真正不同的选择及代价，不直接跳到命名/口号 |
| 执行落地 | 人工批准后服从方向，不重写战略、不带回废案 |
| 模型切换 | Codex/Claude 读取相同当前事实、决定和证据 |
| 品牌质量匿名评审 | 达到 M0 预设的战略锋利度、自然中文、消费者真实、记忆性和产品咬合度基线 |

## 一票否决

以下七项适用于 M0-M2 的每个模型、协议和版本；任一发生即阻断里程碑：

1. 虚构产品事实。
2. 把讨论升级成决定。
3. 把暂定日期写成死线。
4. 把过期方案当成当前方向。
5. 重要结论不能回到证据。
6. 未经人确认自动改变项目状态。
7. 在探索模式下强行制造唯一答案。
