# 两条需求线拆分说明

> 2026-07-22 更新：品牌项目协作已经批准使用公司服务器，但不等于把 FoxWork 的全部团队文件同步、NAS、个人工作区和 PPT 合版需求合入本项目。两条需求线仍分开；服务器只接入经项目授权的资料和业务状态。

需求依据：仓库根目录 `20260713_品牌AI长期项目协作系统_场景化需求澄清与产品反馈_v0.1(1).md`。

## 决策摘要

当前必须把两个真实但不同的问题拆开管理：

| 需求线 | 产品问题 | 第一用户 | 当前状态 | 当前技术形态 |
|:---|:---|:---|:---|:---|
| A. 品牌 AI 长期项目协作 | AI 失忆、会议误读、状态混乱、策略探索被工程化、多模型重复解释 | Fox/试点团队 | **CURRENT / P0** | 公司定制 OpenWork 唯一客户端、Brand Project OS Service、统一 Task Packet/MCP/Skills |
| B. FoxWork 团队文件协作 | 多人文件同步、NAS、个人工作区、版本、权限、远程访问、PPT 合版 | 团队 | **future-candidate** | 中央同步、NAS 或服务器能力，需独立需求确认 |

只有需求线 B 的全量团队文件协作继续使用以下旧候选标签：

- `future-candidate`
- `not-approved-for-current-mvp`
- `review-after-hongri-pilot`

这三个标签不再适用于需求线 A 的 Brand Project OS 服务器。

## 需求线 A：品牌 AI 长期项目协作

### 产品定义

产品首先是长期品牌营销项目的“项目状态与品牌认知协作层”：让 Fox 和不同 AI 都能知道什么是原始证据、事实、观点、假设、选项、倾向、决定、约束和开放问题；知道项目当前做到哪里、本轮是探索还是执行，以及重要结论可以回到哪条原话。

它不是普通项目管理软件、通用企业知识库、纯 RAG、Worktile/Teambition 替代品，也不是先建设完整基础设施再寻找使用场景的平台。

### 当前六个场景

| 场景 | 当前问题 | MVP 必须证明 |
|:---|:---|:---|
| 1. 会议语义不等于正式状态 | “再试一下”“不要太像”“最好月底有一版”会被误写成决定、红线或死线 | 先识别探索/评估/决策/同步模式；保存原话；未经 Fox 确认不升级状态 |
| 2. 文件都在但版本错误 | 语义检索可能命中过期方案并自信回答 | 先读当前状态和本轮任务，再按关系查证据；废案只用于复盘、排重或风险检查 |
| 3. 品牌策略不能直接计算 | 系统过早要求唯一目标、完整字段和确定答案 | 探索协议保留矛盾、发展真正不同的战略选择并说明代价；只有显式切换后才进入执行规格 |
| 4. 新会议不应重写历史 | 全量重总结会覆盖旧决定、重复入库和漂移原话 | 只生成增量变化建议、冲突和证据；Fox 确认后才更新当前状态 |
| 5. 换模型不应重讲项目 | Codex、Claude 等各自依赖聊天记忆，形成不同事实版本 | 所有模型读取同一版本的 Task Packet、当前状态、决定、证据、开放问题和工作模式 |
| 6. 先证明 Fox 提效 | 团队基础设施抢在个人价值验证之前 | 鸿日连续真实使用证明少解释、少误读、可回源、多模型一致和品牌质量提升 |

### 两种工作协议

探索协议与执行规格是两个独立契约，不能只靠 Prompt 中一句话切换：

| 协议 | 使用条件 | AI 应做 | AI 禁止 |
|:---|:---|:---|:---|
| 探索协议 | 研究、洞察、策略、创意方向 | 保留矛盾、重构问题、提出假设、发展不同战略领地、说明取舍和证据缺口 | 自行收口、把倾向升级为决定、直接跳到命名口号、用完整交付掩盖策略缺口 |
| 执行规格 | Fox 已批准方向后的命名、文案、PPT 和物料 | 严格服从批准事实、方向、格式、禁区和验收标准 | 重写战略、重新引入废案、把未批准候选混入交付 |

模式切换必须由 Fox 明确触发并记录，AI 不得以“信息已经足够”为理由自行切换。

### 当前最小数据语义

CURRENT 阶段优先表达：

- 项目阶段、会议、发言人、决定人、原始资料；
- 事实、观点、假设、选项、方向倾向、决定、约束；
- 开放问题、行动、战略领地、方案和交付物；
- `提出于`、`来源于`、`支持`、`反对`、`与之冲突`、`适用于`、`被谁批准`、`替代了`、`依赖于`、`回答了哪个问题`、`仍待确认`。

CURRENT 不要求独立图数据库。SQLite 或同等轻量结构化存储只要能稳定表达关系、版本、来源和增量变更即可。

## 需求线 B：FoxWork 团队文件协作

### 独立问题域

该线关注的是团队文件协作而非品牌认知：

- NAS 和中央文件同步；
- 多成员个人工作区与共享区；
- 文件锁、版本、冲突和权限；
- 远程访问、断网恢复和大文件传输；
- PPT 合版、审阅和团队交付流程；
- 服务器稳定性、账户、审计、备份和灾备。

### 为什么当前不能合并

1. 两条线第一用户不同：A 是 Fox，B 是团队。
2. 第一价值指标不同：A 看 AI 是否理解正确和品牌产出是否变好；B 看同步一致性、并发和可用性。
3. 数据模型不同：A 的核心是证据、观点、决定、开放问题和工作模式；B 的核心是文件、成员、权限、锁和同步。
4. 验证成本不同：A 可用本地轻量原型验证；B 需要生产服务器、账户、安全和运维投入。
5. 过早合并会让 PostgreSQL、S3、OIDC、HA 和灾备成为 A 的前置条件，延迟真正的产品价值验证。

### 未来可能共享的能力

鸿日试点通过后，可重新评估：

- 稳定 ID、哈希和原始文件版本；
- 当前状态与变更提案 Schema；
- Task Packet 和 MCP/CLI 接口；
- OpenWork 桌面壳中的文件与 AI 工作入口；
- 审计、权限和数据分级。

共享必须通过版本化端口发生，不能默认把团队文件系统变成品牌业务真相源。

## 防止再次混线的规则

- 活动文档、验收和开发任务必须回答“这是否支持品牌项目核心旅程，或保障团队一致性与稳定性”。
- 服务器、团队账户、企业权限、PITR、RPO/RTO 和负载均衡按 Phase 2-4 实施，不得再写成未批准候选。
- FoxWork 全量文件协作仍单列，不能因为共用服务器就自动进入 Brand Project OS 范围。
- 不因未来可能复用，就把需求线 B 的模块列为需求线 A 的 MVP 依赖。
- BrandBench 和鸿日黄金用例必须在实现前建立，不得后置为生产验收阶段。
- 是否团队化必须在鸿日真实试点后重新由 Fox 决定，技术方案不得替人预先批准。
