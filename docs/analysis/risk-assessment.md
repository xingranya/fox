> 基线：2026-07-24  
> 当前状态：F3.2 Den 采用门已通过，F3.3 进行中

## 单账号与控制面风险

### 源码和许可

Den 代码位于上游 `ee/**`，使用 FSL-1.1-MIT。F3.2 已证明内部可构建、自托管和修改，但这不等于 OSI 开源，也不允许把它包装成面向外部的 OpenWork 竞争服务。内部 FoxWork 发行仍需保留许可和版权。

### 身份联邦

Den 已有 OAuth Provider、OIDC Discovery 和基础 scopes，但 Brand OS 专用 audience、组织声明与撤权联动尚未完成。F3.5 之前只能说“通用 OIDC 基线和 Den 能力均已验证”，不能说“统一登录已经生产闭环”。

### 账号恢复

Den MySQL 恢复与 Brand OS PostgreSQL 恢复可能选择不同时间点。恢复后必须对账账号、组织、成员版本、稳定身份绑定、项目映射和撤权；不一致时默认拒绝扩大权限。不能让 Brand OS 用邮箱猜测恢复后的 Den 用户。

## 数据与隐私风险

| 风险 | 控制 |
|:---|:---|
| 原件被改写或覆盖 | SHA-256、S3 VersionId、ACTIVE-only、同名新版本和延迟墓碑 |
| Token/模型密钥泄露 | 系统钥匙串/密钥服务、日志脱敏、Renderer 隔离、短期令牌 |
| 跨项目读取 | 应用 RBAC + 保密级别 + RLS + MCP 项目固定 |
| 服务账号冒充人 | 独立主体类型、人工路由隔离、无 `PROPOSAL_REVIEW` |
| 本机资料被服务器扫描 | FoxWork 本机授权、出站连接、真实路径和目录边界 |
| 自托管误判为不外发 | 模型、插件、HTTP、转录和 Embedding 按实际提供商审计 |
| 派生数据污染当前状态 | 派生水位、回权威复核、删除重建、只回传 Proposal |

## 部署与可靠性风险

- F3.2 的临时 Den 服务不是生产部署；F3.3 必须证明重复部署、密钥、迁移、备份、恢复和升级。
- PostgreSQL/S3 F2.10 逻辑恢复不是生产 PITR；F4.6 必须使用托管服务和独立备份域演练。
- Den MySQL 的 RPO/RTO 还未批准；在 F4.8 前不能引用 PostgreSQL 目标替代。
- 单节点小团队档允许维护窗口，但必须有可恢复备份和回滚；不能用“人少”跳过审计、撤权和恢复。
- Den、Brand OS、数据库、对象存储和 Worker 应分开报告健康，避免一个可选组件拖垮核心就绪。

## 供应链与客户端风险

- OpenWork App、Desktop、Server、Orchestrator 之间共享类型和 OpenCode SDK，升级可能跨包破坏。
- Electron 的 IPC、PTY、文件、深链、外部导航和更新面都属于高权限攻击面。
- 全量中文不能只覆盖源码可见文本，还要扫描运行时错误、打包资源、上游链接和未解析键。
- FSL/MIT 归属、第三方依赖、镜像来源、桌面签名和更新清单必须可审计。
- FoxWork 是唯一客户端，因此客户端故障不能靠第二套 Web 兜底；需要灰度、回滚和带水位只读故障态。

## 风险处置顺序

1. F3.3 先消除 Den/远程 Worker 生产部署、备份、恢复和升级未知项。
2. F3.4-F3.6 解决第二套登录、自助注册、员工端/管理员后台全中文、token audience、撤权和项目映射。
3. F3.7-F3.13 再开放真实资料、多媒体处理、本机桥接和公司能力目录。
4. F3.14-F3.18 最后接入 Dify 与可选组件，避免掩盖核心缺口。
5. F3.19 通过前不接真实鸿喜达资料或同步阶段 Wiki；F4.10 通过前不宣称生产上线。

## 治理风险

- 活动任务数固定为 56；历史 49 项只能出现在明确的 rescope 追溯中。
- ADR-0007 是 Den 采用的当前决策，旧“不采用 Den”文案不得重新进入活动计划。
- 业务事实只进入权威事件、证据和 Task Packet，不进入 `AGENTS.md`、Skill 或模型对话记忆。
- 文档中“已完成”“已验证”“内部目标”“待批准”必须分开，不能用计划语气虚报现场状态。
