# 风险评估

## S.U.P.E.R 架构健康摘要

当前没有代码，以下评估表示团队服务器架构的实施准备度，而非运行质量。

| 原则 | 状态 | 主要发现 | 优先级 |
|:---|:---|:---|:---|
| S 单一职责 | 黄 | 领域、应用、Worker 和外部适配器边界已提出，但 Nubase、Open Notebook、Dify 等宽平台可能吞并领域职责 | 高 |
| U 单向流 | 红 | PostgreSQL 单一权威方向已明确，事务、Outbox、回调复核和索引水位尚未实现 | 最高 |
| P 接口优先 | 红 | CanonicalStore、ObjectStore、Search、Research、AIWorkflow、ApprovalWorkflow、Model 端口尚未落地 | 最高 |
| E 环境无关 | 黄 | 目标已从个人本机改为服务器，但 TLS、密钥、迁移、备份、可观测和故障域约束尚未固化 | 高 |
| R 可替换性 | 红 | 若客户端直连组件、泄漏上游 Schema 或把 Dify/FlowLong 状态当正式状态，替换成本会快速上升 | 最高 |

整体健康：0/5 完全健康。分析已经确定正确边界，但必须先把端口、Schema、事务约束和恢复路径变成可执行契约，再引入候选组件。

### 最高风险热点

1. PostgreSQL、Notebook、Memory、索引、Dify 和 FlowLong 同时保存“当前状态”。
2. 团队成员并发编辑或审批时发生静默覆盖、重复提交或双重批准。
3. 数据库提交成功而索引、通知或外部流程派发失败，且没有可重试的 Outbox。
4. Web、MCP、CLI、Skills 各自实现权限和业务规则，产生旁路写入。
5. 单服务器、同盘备份或未经恢复演练的“备份”被误认为高可用。
6. Open Notebook、Nubase、Dify 或 FlowLong 的内部对象泄漏进领域模型。
7. AI Memory、摘要、相似检索或工作流输出被升级为正式事实。
8. Fox 从“让 AI 准备工作”转变成“逐条维护基础设施和审批所有低风险候选”。

## 高严重度风险说明

### 单一权威与一致性

“内容真相”和“状态真相”必须分开：对象存储中的不可变原件证明说过什么；PostgreSQL 中的人工批准事件证明现在正式是什么。当前状态投影、搜索索引、Notebook、Memory、Dify 和 FlowLong 都只能是投影、候选或协调层。

正式写入的最小原子单元是“领域变更 + 审计事件 + Outbox”。索引和外部流程允许最终一致，但界面必须显示索引水位或同步状态。任何客户端都通过统一应用服务写入，禁止数据库直连和第三方回调直写正式表。

### 服务可用性

Docker Compose 能降低首版运维复杂度，但不会自动带来高可用。单台 VPS 同时运行反向代理、API、Worker、PostgreSQL 和 MinIO 时，主机、磁盘和网络都是共同故障点。首发可以接受已声明的单点，但不得把同机备份计入灾难恢复。

团队生产基线建议：月度可用性 `99.5%`、`RPO <= 5 分钟`、`RTO <= 1 小时`。达到更高业务依赖后再升级为两个 API 实例、独立 Worker、托管或高可用 PostgreSQL、外部 S3 和负载均衡；详细分档见 [部署拓扑评估](deployment-topology-evaluation.md)。

### 隐私外发

自托管不等于不外发。模型、Embedding、STT、TTS、网页抓取、对象存储、Dify 插件和 Open Notebook 提供商均需进入出口审计。服务端部署后，原有“只绑定 localhost”假设失效，必须使用 TLS、最小权限凭据、密钥轮换和出口控制。

### 审批负担

审批队列按状态风险分级。每场会议目标不超过 8 张卡；个人观点、重复候选和低风险内容自动归档。Dify 可以聚类、归纳和生成建议，FlowLong 可以路由复杂人工任务，但两者都不能扩大需要 Fox 手工确认的范围。

## 服务目标与告警风险

| 指标 | 团队生产建议 | 告警条件 |
|:---|:---|:---|
| 月度应用可用性 | `>= 99.5%` | 预算消耗速率超过 2 倍或连续 5 分钟不可用 |
| 读取 API 延迟 | `p95 <= 500ms`，不含模型生成 | 连续 10 分钟超标 |
| 普通写入延迟 | `p95 <= 1s`，不含文件上传和外部流程 | 连续 10 分钟超标 |
| Outbox 新鲜度 | `95% <= 60s`，`99% <= 5min` | 最老未处理事件超过 5 分钟 |
| 权威回源完整率 | `100%` | 任一正式结论无法回源立即告警 |

| 服务 RTO | `<= 1 小时` | 恢复演练或真实恢复超时 |

## 项目治理风险

- 业务事实不得写入 AGENTS、CLAUDE、Skill 文件或模型对话记忆。
- 规划和执行状态只能进入 `docs/progress/MASTER.md` 与阶段文件。
- 技术决策在实施阶段建立 ADR，记录决策、替代项、证据和复审条件。
- 每个依赖必须登记固定版本、镜像摘要、许可证、数据范围、升级责任和退出路径。
- 生产数据库迁移必须显式审批和执行，不允许应用启动时自动修改 Schema。
- 没有用户明确批准时，不创建仓库内 Memory 文件。

## 兼容性关注

- Zvec 列出的 macOS 支持为 ARM64；服务器 Linux 目标需要独立 POC，并始终保留 PostgreSQL FTS 降级。
- Zvec 单写多读，需要独立串行索引 Worker 或按索引分区保证唯一写者。
- Open Notebook 要求 Python 3.11 到 3.12、SurrealDB 和多个 AI 依赖；只复用 `content-core` 可显著减轻依赖。
- Nubase 引入 Java、PostgreSQL、Redis、Node 和 Docker，且当前 early-stage、缺少 PITR/HA，不能作为透明的“数据库替换”。
- Dify 自托管栈包含自身数据库、Redis、Worker、插件和模型配置；必须与 Brand OS 数据库和权限隔离。
- Dify 使用基于 Apache-2.0 修改的许可证：源码多租户运营需书面授权，使用其前端时不得移除或修改 Logo/版权信息。
- FlowLong 引入 Java 17、MyBatis Plus、MySQL 和独立设计器，其附加许可需单独法律审查。

## 硬性门槛

- 生产正式业务写入只进入一个 PostgreSQL 权威库。
- 未经人工批准的状态升级次数必须为 0。
- 已批准事实、决定和约束的证据链完整率必须为 100%。
- 删除检索、Notebook、Memory、Dify 和 FlowLong 协调数据后，权威状态仍完整可读。
- 备份只有在独立环境完成 PITR、对象恢复和派生层重建后才算有效。
- 任一上游升级必须先通过 V5 金标数据、契约测试和 BrandBench 回归。
- 未通过许可门时，不得把 Dify 用作源码多租户服务，也不得嵌入或分发 FlowLong。
