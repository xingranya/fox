## 六个场景的失败判定

### 1. 会议解释失败

任何未经 Fox 确认的讨论、偏好、假设、选项、方向倾向或暂定日期进入正式状态，即为失败。模型置信度高、多人重复提及和措辞强烈都不能替代明确决定人、决定动作、适用范围与人工确认。

### 2. 当前版本失败

回答引用过期方案、废案、被替代事实，或无法说明内容为何当前有效，即为失败。向量相似度、文件更新时间和模型偏好都不能单独决定权威性。

### 3. 策略探索失败

探索任务只给一个“正确答案”、直接输出口号、选择之间没有实质差异或不说明代价，即为失败。结构完整不能补偿策略选择缺失。

### 4. 增量会议失败

新会议导致全历史被重写、重复项无去重、冲突被静默合并、原话被摘要替代或用户看不出本次变化，即为失败。

### 5. 多模型失败

Codex 与 Claude 在同一任务中读取了不同事实、决定、工作模式、证据集合或状态版本，即为系统失败，而不是模型风格差异。

### 6. 个人价值失败

Fox 连续真实使用后仍需重复解释项目、频繁纠正状态、无法快速回源，或品牌产出质量不如现有方式，即使功能和测试全部通过也判定 MVP 失败。

## 一票否决错误

- 虚构产品或项目事实；
- 把讨论升级成决定；
- 把暂定日期写成死线；
- 把过期方案当成当前方向；
- 重要结论不能回到证据；
- 未经 Fox 确认自动改变项目状态；
- 在探索协议下强行制造唯一答案。

任一发生时，停止扩展功能，先修订分类、Task Packet、模式协议或回源链，并加入永久回归样本。

## 数据与隐私风险

| 风险 | CURRENT 控制 |
|:---|:---|
| 原件误改 | 本地原件只读；写入独立元数据/派生目录；修改前后哈希校验 |
| 敏感资料外发 | 每次模型调用记录提供商、任务和资料范围；默认只传最小证据；敏感级别可禁止外部模型 |
| Prompt/日志泄漏 | 日志不保存完整原文、密钥和不必要 Prompt；调试导出需人工触发 |
| 模型聊天成为事实源 | 聊天只保存运行历史；当前状态与 Task Packet 从本地核心生成 |
| 删除与覆盖 | CURRENT 不自动删除原件；状态使用替代/废弃关系，不物理覆盖历史 |

“本地运行”不等于“数据不外发”。Codex、Claude、Embedding、转写或网页抓取只要调用外部服务，仍需明确数据范围和 Fox 授权。

## OpenWork 风险边界

OpenWork 已确定为唯一员工客户端基础，但不能改变上述风险模型：

- OpenWork Server、OpenCode Session、Permission、SQLite/JSON 只属运行态；
- OpenCode Tool Permission 不等于 Fox 的业务批准；
- Renderer Token、Electron 沙箱、IPC、文件/PTY、更新源和许可证风险在任何本地试用中仍需控制；
- F1.9 已验证离线出口、Electron 安全、品牌、单安装包和产物；企业身份、远程 Worker 与公司更新源仍在后续阶段验收；
- 清除 OpenWork/OpenCode 运行数据后，正式状态、证据、Task Packet 和模型入口必须仍可恢复使用。

## Phase 3 适配器风险

Zvec、Nubase、Open Notebook、FlowLong 和 Dify 均通过可替换端口接入：

- 不进入核心联网旅程的必需链路；
- 不保存当前状态或批准；
- 不让内部 Schema 进入本地核心；
- 必须有无需该组件的直接基线；
- 每项单独做许可、隐私、质量、稳定性和退出 POC。

## Phase 2-4 团队风险登记

以下风险进入活动计划：团队账户与撤权、跨项目隔离、多用户并发、PostgreSQL 事务、S3 对账、Outbox、OIDC/MFA/RLS、HA、PITR、RPO/RTO、桌面签名更新和灾备。

这些能力不能反向成为 F1.9/F1.10 的运行前置；进入 Phase 2 后则是服务器阶段的硬验收。

## 治理风险

- 本轮已消除 analysis 与 plan/progress 的范围冲突；后续若把服务器重新写成未批准候选，或把它反向设为 Phase 1 前置，都属于范围回归。
- 开发仓库 `AGENTS.md` 不能替代运行时品牌 Agent 宪法、工作模式协议和鸿日项目规则。
- 稳定业务事实不进入 Skill、Prompt 或模型对话记忆；它们只通过当前状态和 Task Packet 提供。
- 每次真实错误必须加入黄金用例并说明修订了哪个规则，避免只在 Prompt 中临时补丁。
