# F1.7 Task Packet 与 Agent 运行留痕

## 现在怎么开始一项 AI 工作

Fox 先登记任务目标、角色、工作模式、交付物和上下文范围。系统再按这份任务定义生成 Task Packet。模型不能临时给自己换角色，也不能把 `work_mode` 作为普通参数改掉。

任务换模式时，Fox 需要写明从什么模式切到什么模式、为什么切、影响哪项任务以及切换时的项目状态版本。旧 Task Packet 不会被改写，新运行会使用新的任务版本。

## Task Packet 里有什么

当前实现使用 `task-packet.v2`，默认装配 L0-L2：

- L0 是本轮任务、品牌角色、工作模式、可用工具、模型范围、输出要求和一票否决项。
- L1 是本轮明确引用的当前决定、开放问题、其他状态、资料缺口和冲突。
- L2 是这些内容直接使用的会议原话或固定来源版本。

L3 原件和 L4 历史默认不加载。需要看原文时按 `evidence://sha256/...` 打开；只有复盘、排重、冲突检查或风险检查才加载历史与废案。

上下文不是“项目越多越好”。任务没有明确引用的决定不会进入 Packet；已被替代、过期、尚未生效或找不到的内容会被排除，并在 `known_gaps` 里说明原因。证据无法解析时也保留“未确认”提示，不按相似内容猜来源。

## 角色和模式

首版固定四个品牌角色：品牌策略、品牌研究、创意协作和执行协作。每个角色都有必须做和不能做的事。

工作模式同样固定为探索、评估、决策准备和执行。AI 可以建议切换，但 `runtime_may_apply_switch` 永远是 `false`，切换权属于 Fox。切换事件使用 `runtime-mode-switch.v2`，项目状态版本和任务修订号都是整数。

这套限制与 Proposal 审批分开。工作模式切换不会批准事实、决定或约束，也不会增加正式项目状态版本。

## 每次运行留下什么

`runtime-run.v1` 会记录：

- Task Packet ID、版本和内容哈希；
- 项目状态版本和任务版本；
- 品牌角色、工作模式和五项协议版本；
- Agent Runtime、模型及各自版本；
- 谁发起运行、何时发起和幂等键。

角色和模式只能从已保存的 Task Packet 复制，运行请求没有覆盖这两个字段的入口。模型切换时，Codex、Claude 或 OpenCode 可以换，但本轮事实、证据和模式不会跟着变。

## 存储和备份

SQLite Schema 升级到 v7。运行任务、模式切换、Task Packet 和 Agent 运行都属于派生运行数据，不进入正式事件流，也不能改变当前业务状态。

在线备份清单升级到 `sqlite-backup.v6`，新增运行任务、模式切换、Packet、运行数量和运行数据摘要。v1-v5 清单及真实 SQLite v2-v6 仍可恢复。

## 验证

F1.7 完成时，133 项测试和 5 组旧 Schema 子测试通过。覆盖最小上下文、无关项隔离、过期项排除、未确认证据、人工模式切换、旧 Packet 保留、Packet 哈希、运行幂等、模型版本绑定、v7 迁移和运行数据备份恢复。

真实鸿日数据库副本已从 v3 迁移到 v7，并通过 `quick_check`。迁移前后项目版本都是 2，事件 2 条，正式状态和 Proposal 都是 0；四张运行表也保持为空。本轮没有为鸿日创建新的决定、开放问题或其他正式业务事实。
