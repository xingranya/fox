# F1.2 SQLite 权威状态

## 当前结果

本地核心已经能保存项目、来源版本、分类候选、Proposal、关系、人工动作、只追加事件和当前状态投影。SQLite 是当前适配器，不是领域规则本身；稳定命令边界见 `contracts/phase1/canonical-store.json` 和 `CanonicalStorePort`。

## 什么能改变当前状态

AI、工作流和系统可以登记候选、Proposal 与工作层关系，但这些记录不会进入 `state_items`。只有配置为人工评审人的 Fox 能执行批准、修改后批准或驳回。

批准时，以下内容在同一个 SQLite 事务内完成：

1. 追加 `PROPOSAL_APPROVED` 事件；
2. 更新 Proposal 状态；
3. 写入人工动作及理由；
4. 更新当前状态投影；
5. 增加项目版本并保存幂等结果。

任一步失败，整条命令回滚。Tool Permission、AI 身份、模型身份和 OpenWork 会话都不能代替这次人工动作。

## 数据表

| 表 | 保存内容 | 权威性 |
|:---|:---|:---|
| `projects` | 项目与当前版本 | 权威版本入口 |
| `commands` | 幂等键、请求摘要和原结果 | 防重复写与同键异义 |
| `events` | 只追加领域事件 | 正式状态形成记录 |
| `sources` | 来源 ID、SHA-256、路径、角色、保密与状态 | 原件版本索引，不保存无来源正文 |
| `classification_candidates` | 带来源哈希和定位的分类候选 | 工作层 |
| `proposals` / `proposal_evidence` | 变化建议与证据 | 未确认前不权威 |
| `relations` | 带证据的工作层关系 | 未确认关系 |
| `human_actions` | Fox 的动作、理由、旧新值、证据和版本 | 人工审计记录 |
| `state_items` | 由批准事件生成的当前投影 | 可从事件重建 |

分类候选使用复合外键绑定项目、来源 ID 和来源哈希。即使绕过应用层直接写 SQLite，也不能把候选悄悄指向另一个文件版本。

## 写入规则

- 每条写请求必须带 `idempotency_key` 和 `expected_version`。
- 同一幂等键和同一请求返回原结果，不重复追加事件。
- 同一幂等键换了请求内容直接冲突。
- 预期版本过期时返回当前版本，不做静默覆盖。
- 一个命令当前只追加一个项目版本事件；项目版本和聚合版本都有唯一约束。
- 数据库文件权限固定为 `0600`，所在 `.fox/state` 目录为 `0700`。

## 迁移

当前 Schema 版本为 2：版本 1 建表和约束，版本 2 建读取索引。每版迁移保存 SHA-256；已应用迁移内容发生变化时程序拒绝启动。单版迁移中任一 SQL 失败，整版回滚。

当前不提供破坏性降级迁移。以后需要改变既有表语义时新增迁移版本，并先完成在线备份；不能改写已经登记的迁移。

## 投影重建

`state_items` 只从已配置人工评审人产生的 `PROPOSAL_APPROVED` 事件重建。删除或损坏投影不会改变事件历史；重建时如果发现批准事件不是人工产生，程序停止。

## 备份与恢复

活跃 SQLite 不允许使用普通文件复制。`SQLiteBackupService` 调用 SQLite 在线备份 API 生成一致快照，清单记录：

- 数据库 SHA-256 与大小；
- Schema 版本；
- 事件、Proposal 和人工动作数量；
- 各项目版本；
- 当前投影摘要。

恢复前验证快照哈希和 `quick_check`，恢复到新数据库后再次对账清单。普通目录备份一旦遇到 `.db`、WAL 或 SHM 文件会直接停止。
