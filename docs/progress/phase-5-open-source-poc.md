# Phase 5：五个外部组件隔离 POC

**目标**：用统一契约、基准和故障场景验证 Zvec、Open Notebook、Nubase、FlowLong、Dify 的真实价值，不产生第二真相源。  
**状态**：未开始；只有 Phase 4 金标旅程通过后才能进入  
**计划窗口**：团队服务器试运行版后约 1-2 周；Nubase 与 FlowLong 受独立门禁约束

## 任务

- [ ] **F5.1：建立统一适配器契约、基准、故障注入与评分表**
  - 优先级：P0
  - 工作量：M
  - 依赖：F4.8
  - Lane：A
  - 测试期望：为搜索、内容处理、平台服务、审批工作流和 AI 工作流定义共享 Contract Test、金标语料、故障注入与降级测试。
  - 治理影响：固定“以契约和证据决策，不以星数或功能数量决策”；所有 POC 使用隔离数据副本、固定版本和可删除环境。
  - 验收：评分覆盖正确性、回源、中文、租户隔离、隐私、性能、恢复、维护、许可、可替换性和总拥有成本；硬淘汰条件可自动判定。
  - 备注：无。

- [ ] **F5.2：执行 Zvec `SearchIndexPort` POC**
  - 优先级：P1
  - 工作量：M
  - 依赖：F5.1、F2.6
  - Lane：B
  - 测试期望：运行 CJK 全文、向量、混合检索、过滤、权限裁剪、单写多读、增量更新、全量重建、进程故障和世代切换测试。
  - 治理影响：Zvec 仅为可重建派生索引；固定单写者与版本；不得接受正式状态写入或成为唯一证据来源。
  - 验收：删除或禁用 Zvec 后主系统仍可用 PostgreSQL 检索与完整回源；只有质量或性能显著优于基线且运维成本可接受时才建议采用。
  - 备注：无。

- [ ] **F5.3：执行 Open Notebook/`content-core` 单向研究 POC**
  - 优先级：P1
  - 工作量：L
  - 依赖：F5.1、F2.3
  - Lane：C
  - 测试期望：运行 PDF、Office、音频、引用映射、REST/MCP、重复处理、外发策略、恶意内容、服务中断和关闭 sidecar 降级测试。
  - 治理影响：明确 `content-core` 或 sidecar 边界；解析、摘要和研究结果只进入候选或派生区，不直接写正式状态。
  - 验收：关闭 Open Notebook 后上传、权威状态和基础回源仍完整；所有派生引用映射回原始稳定 ID；外发受保密级别与人工授权控制。
  - 备注：无。

- [ ] **F5.4：执行 Nubase 单项能力与生产缺口 POC**
  - 优先级：P2
  - 工作量：L
  - 依赖：F5.1、F2.7、F3.2
  - Lane：D
  - 测试期望：运行 PostgreSQL 兼容、RLS、Auth、Storage、备份恢复、迁移退出、服务中断、跨租户攻击和 Memory 权限测试。
  - 治理影响：登记 early-stage、PITR、HA、升级和供应链风险；Nubase Memory 只能保存派生记忆，不能写事实、决定或约束。
  - 验收：POC 不与正式 PostgreSQL 双写；可完整导出并迁回既有端口；恢复目标和租户隔离达到基线后才可能采用，否则明确推迟或拒绝。
  - 备注：团队服务器需求已成立，但 Nubase 仍不得未经验证承担生产真相库。

- [ ] **F5.5：完成 FlowLong 书面许可门与复杂人工审批 POC**
  - 优先级：P2
  - 工作量：L
  - 依赖：F5.1、书面许可门
  - Lane：E
  - 测试期望：许可通过后运行批准、驳回、会签、回调幂等、并发审批、宕机恢复、流程版本迁移和禁用自动批准测试。
  - 治理影响：附加许可是硬门；FlowLong 仅协调复杂流程，权威审批结果仍由领域命令和事件记录；AI 或超时自动批准必须关闭。
  - 验收：未取得书面许可时记录 `BLOCKED` 或 `DEFERRED` 且不引入源码；许可通过后，只有复杂审批收益超过 Java、运维与退出成本时才建议采用。
  - 备注：当前书面许可尚未取得。

- [ ] **F5.6：执行 Dify `AIWorkflowPort` 与直接 Worker A/B**
  - 优先级：P1
  - 工作量：L
  - 依赖：F5.1、F4.1、F3.5
  - Lane：F
  - 测试期望：对同一会议与 Task Packet 运行直接 Worker/Dify A/B，覆盖输出 Schema、引用、幂等、超时、重试、回调伪造、工作流版本、外发审计、服务中断和 NoOp 降级。
  - 治理影响：Dify 只能通过版本化 `AIWorkflowPort` 协调模型与工具；不得直连权威数据库、批准提案或绕过保密和外发策略；许可形态、模型/插件清单和外发路径进入 ADR。
  - 验收：首 POC 限单内部 Workspace、后端 API 和最小 Task Packet；禁用 Dify 后可切回直接 Worker；输出只形成带来源和工作流版本的候选；未经书面授权不做源码多租户、白标或分发。
  - 备注：Dify 不替代领域状态机、Outbox Worker 或人工审批。

- [ ] **F5.7：输出五组件采用、推迟或拒绝 ADR**
  - 优先级：P0
  - 工作量：M
  - 依赖：F5.2、F5.3、F5.4、F5.5、F5.6
  - Lane：A
  - 测试期望：运行文档链接、决策表完整性、POC 证据、硬门和退出演练结果的静态复核；文档任务不新增运行时代码。
  - 治理影响：ADR 是 Phase 6 是否集成外部组件的唯一依据；未触发或被许可阻断的 POC 必须明确记录状态，不能默认视为通过。
  - 验收：每个组件都有角色、数据流、真相边界、版本、许可、SLO 影响、成本、风险、退出方案、升级策略和 `ADOPT`/`DEFER`/`REJECT` 结论。
  - 备注：无。

## 阶段备注

- 六条 POC Lane 使用隔离目录、隔离凭据和脱敏数据副本，不连接正式权威库。
- 所有外部组件必须支持禁用、替换和重建；POC 不得修改领域 Schema，也不得长期双写。
- FlowLong 书面许可未通过时，F5.5 可按门禁记录为阻断或推迟，但 F5.7 必须如实反映，不能伪造测试结论。
