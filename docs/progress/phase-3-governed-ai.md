# Phase 3：团队治理与 AI 统一访问

**目标**：让团队成员和 AI 通过同一服务器身份、权限、审批与审计边界访问工作状态。  
**状态**：未开始  
**计划窗口**：约 2 周，可在 Phase 2 数据契约稳定后分 Lane 推进

## 任务

- [ ] **F3.1：实现 OIDC、MFA、邀请、会话、撤权与服务账号**
  - 优先级：P0
  - 工作量：L
  - 依赖：F1.4、F2.1
  - Lane：A
  - 测试期望：增加 OIDC 回调、声明校验、MFA 策略、邀请过期、会话轮换、全端撤权、禁用用户和服务账号密钥轮换测试。
  - 治理影响：固定身份提供商信任边界、强认证对象、紧急撤权时限和服务账号最小权限规则。
  - 验收：用户只能通过组织批准的身份登录；高权限角色启用 MFA；撤权后现有会话和令牌在约定时间内失效；服务账号有独立身份与审计。
  - 备注：无。

- [ ] **F3.2：实现 RBAC、动作 Scope、保密策略与 RLS 事务上下文**
  - 优先级：P0
  - 工作量：L
  - 依赖：F3.1、F2.1
  - Lane：A
  - 测试期望：增加 Owner、Approver、Contributor、Viewer 权限矩阵，项目 Scope、保密级别、跨租户拒绝、连接复用和权限降级测试。
  - 治理影响：角色只授予能力，不改变事实权威；应用授权和数据库 RLS 必须双重校验，禁止仅靠前端隐藏。
  - 验收：每个请求都将组织、工作区、项目、角色和保密上下文安全注入事务；横向与纵向越权均被拒绝并审计；权限变更即时影响新请求。
  - 备注：无。

- [ ] **F3.3：实现 Proposal 与正式状态迁移策略**
  - 优先级：P0
  - 工作量：M
  - 依赖：F2.2、F3.2
  - Lane：B
  - 测试期望：增加状态矩阵、模糊会议、提示注入、无证据提案、跨项目提案和属性测试。
  - 治理影响：把 AI 只能创建候选提案、不能批准事实或外部承诺的规则落实到领域策略，而非只写在提示词中。
  - 验收：无人工批准事件时正式状态不变；每个提案保留来源、依据、提出者、影响范围和目标版本；非法迁移在领域层统一拒绝。
  - 备注：无。

- [ ] **F3.4：实现批准、修改、驳回、替代、重开与并发审批**
  - 优先级：P0
  - 工作量：L
  - 依赖：F3.3
  - Lane：B
  - 测试期望：增加完整审批生命周期、重复提交、双人并发、版本冲突、批量边界、撤权后审批和替代链测试。
  - 治理影响：固定谁可批准什么、审批理由和证据要求；已批准记录只追加替代或重开事件，不原地覆盖。
  - 验收：批准、修改、驳回、替代和重开均保留操作者、时间、理由、范围、前后版本与证据；并发冲突返回 409 且不会丢失任一提案。
  - 备注：无。

- [ ] **F3.5：实现当前状态、证据服务与最小 Task Packet**
  - 优先级：P0
  - 工作量：M
  - 依赖：F2.5、F3.3
  - Lane：B
  - 测试期望：增加工作模式、权限裁剪、证据缺口、最小上下文、版本兼容、无关历史排除和来源回链测试。
  - 治理影响：固定 Task Packet 版本、最小披露和“状态摘要不是独立真相源”；敏感数据按任务范围裁剪。
  - 验收：输出目标、已批准事实、决定、约束、开放项、任务、证据和禁区；每项结论可回源；未授权或无关历史不进入上下文。
  - 备注：无。

- [ ] **F3.6：实现统一 HTTP/OpenAPI、审计与错误模型**
  - 优先级：P0
  - 工作量：L
  - 依赖：F3.2、F3.4、F3.5
  - Lane：C
  - 测试期望：增加 OpenAPI 快照、鉴权、租户上下文、幂等、409、限流、分页、错误映射、审计关联 ID 和敏感字段脱敏测试。
  - 治理影响：HTTP 是 Web、MCP、CLI 和 Skills 的统一服务契约；固定错误码、审计字段、兼容策略和弃用窗口。
  - 验收：所有入口共享同一应用服务与权限检查；OpenAPI 可生成客户端；请求、身份、动作、结果和关联 ID 可审计，正文与令牌不泄漏到日志。
  - 备注：无。

- [ ] **F3.7：实现 OAuth 保护的远程 MCP 与本地 stdio 代理**
  - 优先级：P0
  - 工作量：M
  - 依赖：F3.6
  - Lane：C
  - 测试期望：增加 HTTPS 连接、身份令牌、Scope、工具 Schema、超时重连、令牌过期、审计、版本协商和 stdio 代理错误测试。
  - 治理影响：MCP 只暴露读取、检索、回源和创建提案能力；批准仍由受控 Web 或明确确认型 CLI 完成。
  - 验收：Codex 等客户端可通过远程 MCP 获取同一团队状态；本地代理不保存长期明文令牌；关闭 MCP 不影响 Web/API；不存在绕过审批的工具。
  - 备注：无。

- [ ] **F3.8：实现 CLI 登录、系统钥匙串、诊断命令与团队 Skills**
  - 优先级：P0
  - 工作量：M
  - 依赖：F3.6
  - Lane：D
  - 测试期望：增加设备登录或授权流程、钥匙串读写、登出清理、`whoami`、`doctor`、网络失败、权限不足和 Skill 契约烟测。
  - 治理影响：Skills 只保存可复用流程，不保存业务事实；CLI 和 Skills 必须连接同一服务器并遵守相同权限与审计。
  - 验收：用户可安全登录、查看当前身份/项目、诊断连接并调用只读或提案命令；配置文件无明文长期密钥；团队 Skill 明确前置读取、回源和提案边界。
  - 备注：无。

## 阶段备注

- Lane A 先建立身份与权限；Lane B 在权限上下文可用后完成提案、审批和 Task Packet。
- F3.6 汇合领域能力后，F3.7 与 F3.8 可并行。
- MCP、CLI、Skills 和 Web 都是同一服务器应用服务的适配器，不各自保存正式状态。
