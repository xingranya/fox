# F2.10 服务器恢复演练与阶段门

## 现在已经做到什么

F2.10 不再把“有备份文件”当成恢复能力。当前实现会从 PostgreSQL 的一致快照生成逻辑备份，再恢复到一个全新的空数据库。恢复结束后，系统逐表核对行数和 SHA-256，重建当前状态与 Proposal 生命周期，并按数据库记录的 S3 VersionId 检查原件。

实现入口：

- `src/brand_os/postgresql_backup.py`
- `src/brand_os/server_recovery.py`
- `contracts/phase2/server-recovery.json`
- `schemas/phase2/server-recovery.schema.json`
- `tests/phase2/test_server_recovery.py`

## PostgreSQL 备份和恢复

备份使用 `pg_export_snapshot()` 固定一个只读一致视图，再让 `pg_dump --snapshot` 导出 custom-format 归档。数据库表摘要和归档来自同一个恢复点，避免先查计数、后导出数据造成口径漂移。

备份清单只保存：

- 数据库 Schema 版本；
- 每张表的行数和整体摘要；
- 归档文件大小、SHA-256 和工具版本；
- 备份 ID、时间和恢复方式。

清单不保存表内容、DSN、密码、Token、项目名称或原件正文。归档文件权限固定为 `0600`。生产环境仍必须把归档放进加密、独立的备份域；文件权限不能代替存储加密。

恢复只允许写入空数据库，并使用 `pg_restore --single-transaction`。不支持原地覆盖，也不会先清空已有数据库。恢复后必须满足：

1. 全部表的行数和摘要与备份点相同；
2. Schema 版本一致；
3. 事件自增序列不落后于现有事件；
4. PostgreSQL 权威存储和对象元数据仓库均通过 `quick_check()`。

## 投影和原件怎么核对

数据库恢复成功后，演练会在恢复库中重新执行：

1. 根据正式 Proposal 事件重建 Proposal 生命周期；
2. 根据人工批准事件重建 `state_items`；
3. 再次计算全库摘要，结果必须和备份点完全相同；
4. 对 PostgreSQL 对象元数据和 S3 版本做一次只读对账；
5. 对所有未物理删除的对象版本复核明确桶、对象键、VersionId、大小和 SHA-256。

当前对象键即使被加了删除标记，只要数据库登记的明确 VersionId 仍保留，原件仍能恢复。若明确版本已被物理删除，系统会记录 `active_object_missing` 并阻断阶段门，不会拿同名文件或最新版本顶替。

## 已覆盖的故障

专项测试覆盖以下情况：

| 故障 | 结果 |
|:---|:---|
| 备份归档被改动 | 恢复前因 SHA-256 或大小不符而停止 |
| 恢复目标已有业务对象 | 拒绝恢复，不清空、不覆盖 |
| 人工批准事件被改成 AI 事件 | 投影重建失败，阶段门停止 |
| ACTIVE 原件的明确 S3 版本缺失 | 对账记录异常并停止，不使用同名对象替代 |
| 当前对象键存在删除标记 | 仍按备份元数据中的 VersionId 读回正确原件 |
| 备份点之后源库继续写入 | 新写入不会混进旧恢复点 |
| 恢复后继续追加事件 | 事件序列正常推进，不发生主键冲突 |

F2.10 专项 6 项、Phase 2 的 133 项测试和 292 项完整回归已经通过。测试只使用本机回环地址上的临时 PostgreSQL 17 与 Moto S3，结束后自动退出；没有读取、上传或迁移鸿日、鸿喜达正式资料。

## 这次演练不能证明什么

本次逻辑备份不是 PostgreSQL PITR，也没有验证公司服务器、云存储、跨账号备份、网络带宽、真实数据量或值班响应。因此：

- 本地恢复耗时只能证明代码链路可执行，不能写成生产 RTO；
- 同一个版本化桶不是独立备份域；
- `pg_dump` 不能代替连续 WAL 归档；
- 高可用、多可用区和自动故障切换仍需 Phase 4 的真实部署和故障演练；
- 恢复点选择、生产切换和 SLO 确认必须由获授权员工完成，AI、MCP、Skill、工作流和服务账号都没有这项权限。

## Fox 已批准的内部目标

Fox 于 2026-07-23 批准采用“小团队托管部署”档位，并把下列数值作为第一批内部试点目标，不作为对外承诺：

| 项目 | 建议目标 |
|:---|:---|
| 部署档位 | 单应用节点 + 托管 PostgreSQL + 开启版本控制的对象存储 + 独立备份域 |
| 核心 API 月可用性 | 不低于 99.5% |
| 读取接口 P95 | 不高于 500 ms |
| 非 AI 写入 P95 | 不高于 1 s |
| Outbox 追平 P95 | 不高于 5 s |
| PostgreSQL RPO | 不高于 5 分钟，必须由连续 WAL/PITR 支撑 |
| 核心服务 RTO | 不高于 60 分钟 |
| 初始容量口径 | 50 个在线会话、10 个并发写请求 |

选择这个档位的原因很直接：首批试点先把恢复、权限和运维做好，不把“两台自建 PostgreSQL”包装成高可用，也不提前引入 Kubernetes。Phase 4 的 F4.5 和 F4.7 必须再用真实部署、数据量和故障时间证明这些目标；测不到就调整目标或延长试点。

F2.10 和 Phase 2 已关闭，可以进入 F3.1。该批准只确认部署档位和内部目标；本地恢复报告中的 `production_pitr_verified=False`、`production_slo_confirmed=False` 必须保留，因为本机夹具没有证明生产 PITR 或生产 SLO。Phase 4 必须用真实托管 PostgreSQL、连续 WAL/PITR、版本化对象存储、独立备份域和故障演练验证；验证前不得把目标写成已达成指标。

## 与 BISHENG 后续规划的关系

BISHENG 继续放在当前 49 项之后。以后若单独批准 BS0-BS3，它需要备份工作流定义、版本化配置和与 Brand Project OS 的引用映射，但不进入正式状态备份，也不能改变恢复点或批准切换。

BISHENG 停机或自身恢复失败时，PostgreSQL 正式状态、原件回源和 OpenWork 人工确认仍要可用。BISHENG 的恢复演练只能建立在本页已经固定的服务器恢复边界上，不能反过来成为 F2.10 或 Phase 3 的前置依赖。
