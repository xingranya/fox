# F2.6 幂等、乐观锁和冲突差异

> 状态：已完成
> 机器契约：`write-consistency.v1`、`write-conflict.v1`
> 任务实现基线：PostgreSQL v9；当前服务器 Schema 已由 F2.7 升至 v10

## 这次解决了什么

服务器原来已经能阻止重复提交和陈旧版本写入，但调用方只能收到通用异常。对 OpenWork 来说，这还不够：员工需要知道请求是已经成功过、幂等键用错了，还是项目在自己操作期间被别人改过。

F2.6 把正式写结果收成三种：

| 结果 | 含义 | 客户端处理 |
|:---|:---|:---|
| `COMMITTED` | 本次请求产生了新事件和新项目版本 | 使用返回的新版本刷新当前页面 |
| `REPLAYED` | 相同幂等键和相同请求已经成功 | 直接复用原结果，不重复提示成功或生成第二条事件 |
| `CONFLICT` | 请求不能按原意提交 | 显示 409 冲突和差异，要求员工刷新后重新确认 |

F2.6 只负责应用层的一致性结果；F2.8 已将这些结果接入版本化路由和 OpenAPI，并稳定映射为 HTTP 409。具体入口见 [F2.8 HTTP API 与 OpenAPI](http-api-and-openapi.md)。

## 三类冲突

| 代码 | 触发条件 | 处理原则 |
|:---|:---|:---|
| `VERSION_MISMATCH` | `expected_version` 与当前项目版本不同 | 不覆盖；返回当前版本、期间事件和正式状态差异 |
| `IDEMPOTENCY_KEY_REUSED` | 同一主体、命令和幂等键绑定了不同请求摘要 | 不执行第二个请求；调用方必须换键或修正请求 |
| `RESOURCE_STATE_CHANGED` | 版本仍是当前，但 Proposal 等资源已不在请求所需状态 | 不伪装成成功；返回资源冲突和当前快照 |

未知异常不会被包装成业务冲突。数据库错误、投影异常和代码错误继续抛出，整笔事务回滚；否则服务端故障会被错误地显示成“请刷新后重试”。

## 差异怎么计算

冲突报告在一个 PostgreSQL `REPEATABLE READ READ ONLY` 快照中完成：

1. 注入 F2.5 已批准的主体、项目、动作和保密级别上限，继续经过强制 RLS。
2. 读取项目当前版本。
3. 从 `PROPOSAL_APPROVED` 事件重建预期版本和当前版本的正式状态。
4. 用当前 `state_items` 投影核对事件重建结果。两者不一致时停止生成报告。
5. 按稳定键计算 `ADDED`、`REMOVED` 和 `MODIFIED`。
6. 返回预期版本之后的脱敏事件元数据；超过上限时给出截断标记和首条未返回事件版本。

报告不复制原始证据正文。状态摘要保存项目版本、项数和规范 JSON 的 SHA-256；事件只带版本、事件/聚合标识、操作者和提交时间。OpenWork 后续需要查看原文时，仍要走证据回源接口。

预期版本大于服务器当前版本时，报告会明确标记基线不可用，不会凭空拼出一个状态。正式状态重复计算结果必须相同；当前投影与批准事件不一致时返回完整性错误，不给客户端一份看似正常的差异。

## 授权与命令

`WriteConsistencyService` 接收 F2.5 的 `AuthorizationDecision`。执行前会核对：

- 授权项目与命令项目相同。
- 授权动作与固定命令映射相同。
- 授权主体与 `CommandContext` 的操作者类型和 ID 相同。

当前正式写映射覆盖来源登记/导入、会议摄取、工作层候选和关系、Proposal 创建/评审/重开。项目初始化走独立引导流程，不混进普通项目写入。

F2.5 已把 MCP 保留为独立服务身份，但现有 `CommandContext` 还没有 MCP Actor 类型。因此 F2.6 明确拒绝把 MCP 冒充 AI 或 Workflow 写入。F3.6 需要先冻结独立 MCP 命令身份，再开放它可获授权的 Proposal 创建入口。

## 验证结果

- 同一请求并发重试 100 次：1 次 `COMMITTED`，99 次 `REPLAYED`，只产生一个领域事件。
- 两个不同请求基于同一版本提交：一个成功，另一个得到 `VERSION_MISMATCH` 和 409 报告。
- 同一幂等键更换载荷：得到 `IDEMPOTENCY_KEY_REUSED`，没有新增事件。
- Proposal 已被处理后再次评审：得到 `RESOURCE_STATE_CHANGED`。
- 陈旧版本报告包含当前版本、批准事件和正式状态新增项；重复计算结果一致。
- 授权项目、动作或操作者不匹配时，在调用存储前拒绝。
- 投影写入异常整笔回滚，不转成假冲突。
- 人为破坏当前投影后，事件核对会阻止冲突报告。

Phase 2 共 93 项测试通过；全量 252 项测试和 19 组子测试通过。测试使用一次性 PostgreSQL 17 集群和 `NOBYPASSRLS` 运行时角色，结束后进程与临时数据库均已清理。没有读取、复制或迁移鸿日/鸿喜达正式资料。
