# ADR-0001：团队服务器作为唯一权威运行面

- 状态：已接受
- 日期：2026-07-13
- 决策人：项目负责人
- 影响范围：生产部署、数据存储、客户端、AI 接入和灾备

## 背景

Brand Project OS 面向团队长期协作。单台个人电脑无法稳定承载多人并发、远程访问、统一权限、持续备份和服务恢复，也会让每个 AI 客户端看到不同状态。

## 决策

1. 生产环境只允许一个团队控制的权威服务端，标准 PostgreSQL 是正式状态、领域事件、审批记录和审计记录的唯一可写数据库。
2. 原始文件进入团队控制的 S3 兼容对象存储，按 SHA-256 内容寻址并开启版本保护。
3. Web/PWA、CLI、MCP、Skills 和后续桌面代理全部调用同一版本化应用 API，不得直接写数据库或对象存储正式区。
4. SQLite 只用于开发测试、导入工具临时库或只读应急快照，不允许与服务器形成双主写入。
5. Zvec、Open Notebook、Nubase Memory、FlowLong、Dify、缓存和模型摘要都是可禁用、可替换的派生层或协调层，不得成为正式状态来源。
6. 首个生产形态采用无状态应用服务加托管 PostgreSQL 和对象存储；不以 Kubernetes 或自建数据库高可用作为首发前提。

## 后果

- 优点：团队成员和 AI 获得同一状态版本；权限、并发、审计、备份和恢复边界明确。
- 成本：首版必须完成身份、RLS、幂等、Outbox、对象上传状态机、监控和恢复演练，开发量高于单机 MVP。
- 限制：离线客户端只能查看允许缓存的数据和保存草稿，不得离线批准或修改正式状态。

## 被拒绝方案

- 每个成员一份 SQLite 后再同步：会引入冲突合并、审批分叉和无法证明的最终状态。
- 单服务器同时自建全部数据库、对象存储和 AI 组件作为正式生产：故障域过于集中，恢复目标难以兑现。
- 直接以 Nubase、Dify、Open Notebook 或 FlowLong 数据库作为领域真相库：组件边界、恢复能力和业务状态语义不满足要求。

## 复审条件

只有在连续恢复演练失败、合规要求变化或标准 PostgreSQL 无法满足经测量的负载时，才允许提出替代方案；替代必须提供无双写迁移、回滚和完整回源证明。
