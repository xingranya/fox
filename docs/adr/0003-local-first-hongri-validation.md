# ADR-0003：本地优先的鸿日单用户验证

- 状态：部分生效；部署拓扑已被 [ADR-0005](0005-single-client-server-authority.md) 取代
- 日期：2026-07-13
- 决策人：Fox
- 第一用户：Fox
- 第一项目：鸿日
- 目标工作空间：`/Users/fox/work`
- 影响范围：Phase 0-1 本地验证、会议解释、人工确认和后续迁移约束
- 后续决策：客户端由 [ADR-0004](0004-openwork-single-client.md) 固定；团队权威服务由 [ADR-0005](0005-single-client-server-authority.md) 批准

> 本 ADR 的会议分类、人工确认、证据回源、Task Packet 和一票否决继续生效。“长期保持本地单用户”和“服务器仅为候选”不再生效。

## 背景

已有规划先后扩展到团队服务器、PostgreSQL、S3、多租户、OIDC、OpenWork 桌面 fork、远程 MCP、Dify、FlowLong 和生产灾备。这些研究有长期价值，但尚未证明产品最核心的假设：AI 能否正确理解鸿日的当前状态，分清讨论与决定，增量处理新会议，回到原始证据，并在 Codex、Claude 等模型之间切换时保持同一项目认知。

品牌工作中的不确定性并非都应被消除。偏好、倾向、假设、选项、目标日期和正式约束具有不同语义；系统若过早收口，会得到技术上严谨但持续误解人的产品。因此，当前必须先以 Fox 的真实工作验证解释质量，再决定团队化和服务器化。

## 决策

1. 当前批准路线是 Fox 本地单用户鸿日 MVP。系统在 `/Users/fox/work` 中运行，不要求生产服务器、团队账户、远程共享或常驻云服务。
2. 本地原始证据采用内容哈希和只读快照保护；SQLite 保存稳定 ID、来源清单、候选分类、只追加事件、Fox 人工审批记录和可重建当前状态投影。二者共同形成当前 MVP 的权威边界。
3. 所有本地入口，包括图形界面、CLI、MCP、Skills 和 Agent Runtime，调用同一领域应用层，不直接修改 SQLite 表或正式证据区。
4. 当前只允许单写入者。多个 AI 可以并行读取或生成候选，但正式变更按 `base_state_version` 串行提交；冲突必须刷新并重新确认，不能静默覆盖。
5. AI 只能生成带证据的 Proposal。Fox 的显式批准、修改或驳回动作才可改变当前正式状态。模型、Skill、MCP、OpenCode Tool Permission 和外部工作流均无批准能力。
6. 会议解释必须先识别会议模式，再分类 `FACT`、`VIEW`、`PREFERENCE`、`HYPOTHESIS`、`OPTION`、`TENDENCY`、`TARGET_DATE`、`DECISION_CANDIDATE`、`CONSTRAINT_CANDIDATE`、`ACTION_CANDIDATE` 和 `OPEN`。任何候选都不得因置信度高而自动升级。
7. 探索协议与执行规格是两个显式工作模式。AI 不得自行判断“信息足够”并从探索切换到执行；切换必须由 Fox 或已批准项目状态明确触发。
8. AI 通过分层 Task Packet 获取最小、当前、相关上下文，先读当前状态和批准记录，再按需回源；不得以随机向量命中或旧聊天记忆代替项目状态。
9. 员工客户端统一采用公司定制版 OpenWork；Brand Project OS 不再另做第二个 Web 或桌面客户端。当前仍先完成本地单用户切片。
10. 本地切片通过后按 ADR-0005 建设服务器权威服务。SQLite 通过一次性校验迁移退出正式写入，不与 PostgreSQL 长期双写。
11. 决定、开放问题和关系通过独立只读端口查询。默认只返回当前未过期内容；被替代、驳回、过期或尚未生效的记录只在显式历史查询中出现。证据引用必须绑定来源版本与 SHA-256，无法解析时明确标记未确认，不按相似度猜测来源。
12. Task Packet 是按任务生成的不可变派生快照。任务角色和初始模式由 Fox 登记，模式切换必须留下人工事件；AI 只能建议切换。每次 Agent 运行必须绑定 Packet 哈希、项目状态版本、任务版本、运行时、模型和协议版本。
13. 本地 CLI 与 stdio MCP 调用同一 `LocalAIService`。项目范围在进程启动时固定，MCP 输入拒绝未声明字段，只开放读取、回源、Proposal 创建和诊断；Codex/Claude 的提供商凭据继续由各自运行时管理，不进入 Brand Project OS。

## 当前架构边界

### 当前必须有

- 鸿日真实资料范围、10-20 个黄金用例和品牌质量评分表。
- 原始资料清单、内容哈希、只读证据快照和原文定位。
- 会议模式识别、分段、分类、去重、冲突检测和增量 Proposal。
- 当前状态、决定、约束、开放问题、行动和证据关系。
- Fox 本地查看、回源、批准、修改和驳回界面。
- 品牌 Agent 宪法、探索/执行协议和项目级规则。
- 面向 Codex、Claude 和其他模型的一致 Task Packet、CLI/MCP 或等价本地入口。
- SQLite 备份、事件重放、投影重建和金标回归。

### 当前明确不要求

- 团队账户、邀请、RBAC、RLS、多租户和并发人工审批。
- PostgreSQL、S3/MinIO、Outbox 服务、WAL/PITR、高可用和公网部署。
- 完整 PWA、移动审批、企业管理面和外部访客。
- OpenWork Server、远程 Orchestrator、托管 Worker、三平台签名与私有更新。
- Zvec、Nubase、Open Notebook、FlowLong、Dify 的生产集成。

## 权威与派生数据

| 数据 | 当前 MVP 角色 | 恢复方式 |
|:---|:---|:---|
| 内容寻址的原始资料快照 | 证明“原文是什么” | 按 SHA-256 校验本地证据区与来源清单 |
| SQLite 事件与 Fox 审批记录 | 证明“当前正式状态如何形成” | 从备份恢复并校验事件序列 |
| 当前状态投影 | 便于读取，不是独立真相 | 从事件和审批记录重建 |
| FTS/向量索引、摘要、模型运行、会话 | 派生或运行态 | 删除后从权威数据重建或重新运行 |
| OpenWork/OpenCode 本地状态 | 客户端与 Agent 运行态 | 删除后不影响鸿日正式状态 |

## 验收标准

1. 新 AI 不读取旧聊天，也能通过 Task Packet 正确说明鸿日当前阶段、目标、已批准决定、开放问题、本轮模式和证据来源。
2. 含讨论、偏好、倾向和暂定日期的会议不会产生非法 `DECISION`、`CONSTRAINT` 或硬截止。
3. 新会议只生成增量变化建议，不重写全部历史；重复、冲突和被替代内容可见。
4. 重要结论能返回原始文件或会议、原话、说话人、时间、适用范围和内容哈希；证据不足时明确标记未确认。
5. 探索任务给出真正不同的战略选择及代价；执行任务服从已批准方向，不带回废案或重写战略。
6. Codex、Claude 或其他模型读取相同 Task Packet 后，事实、决定和证据一致，差异主要来自能力与表达。
7. Fox 在真实工作中减少重复解释和状态混乱，品牌策略与文案候选通过匿名或盲评基线。
8. 删除派生索引、模型会话或 OpenWork 状态后，当前状态与证据链仍可恢复。

### 一票否决

- 虚构产品事实或证据。
- 把观点、偏好、假设、选项或倾向升级为决定/约束。
- 把暂定日期、内部目标或评审节点写成外部硬截止。
- 把过期方案当成当前方向。
- 重要结论无法回到原始证据。
- 未经 Fox 确认自动改变项目状态。
- 在探索模式下强行形成唯一答案。

## 已批准迁移约束

本地实现必须保留稳定项目/来源/陈述/事件/Proposal ID、Schema 版本、内容哈希、`base_state_version` 和明确端口。按 ADR-0005 进入团队服务器时，只允许经过校验的一次性单向迁移：导入 PostgreSQL/S3、对账、冻结本地写入、切换权威入口并保留只读归档。服务器化不得要求重新解释鸿日历史，也不得形成长期双主。

## 复审条件

- 鸿日 Phase 0 黄金测试与本地纵向切片未通过时，优先修正解释协议和产品范围，不扩基础设施。
- F1.10 通过后按 ADR-0005 进入服务器权威基础，不再重复复审是否服务器化或是否另做第二个员工客户端。
- 团队规模、真实负载、恢复演练和外发范围决定服务器部署档位与高可用强度，不能用架构设想代替测量。
