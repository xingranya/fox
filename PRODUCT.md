# Product

## Register

product

## 产品定义

Brand Project OS 是面向长期品牌营销项目的“项目状态与品牌认知协作层”。它让 Fox 与不同 AI 在开始一项工作时，都能准确知道：

- 什么是可回源的事实；
- 什么只是观点、偏好、假设、选项或方向倾向；
- 哪些决定和约束已经由人确认；
- 项目当前处于什么阶段，本轮要完成什么；
- 当前是在探索、评估、决策还是执行；
- 新资料相对当前状态改变了什么；
- 每项重要结论可以回到哪份原件、哪场会议和哪句原话。

它不是普通项目管理软件、通用企业知识库、全量文件向量化 RAG，也不是先建设完整团队基础设施再寻找业务价值的平台。

## 当前验证边界

| 项目项 | 当前结论 |
|:---|:---|
| 第一用户 | Fox |
| 第一验证项目 | 鸿日 |
| 当前形态 | 本地优先、单用户、轻量存储、桌面或本地界面 |
| 当前目标 | 先证明 AI 理解正确、减少重复解释、减少状态混乱并提升品牌产出 |
| 当前非目标 | 团队账户、生产服务器、PostgreSQL、S3、完整 PWA、企业权限、高可用与灾备 |
| 复审门 | 鸿日连续真实工作验证通过后，才决定是否进入团队服务器化 |

FoxWork 团队文件协作是独立需求线，解决 NAS、多人同步、版本、权限、远程访问和 PPT 合版。它未来可能与本产品共享基础能力，但不得提前合并，也不得把它的服务器需求写成当前 MVP 的既定目标。

## 产品目的

当前产品必须同时证明两种能力：

1. **可靠性**：事实、证据、状态、版本和变更不混乱，重要结论可回源，AI 不能擅自改变正式状态。
2. **品牌工作能力**：保留有价值的不确定性，提出真正不同的策略选择，说明取舍，并在已批准方向后稳定执行。

成功不是“上传了多少文件”或“接了多少模型”，而是 Fox 在真实鸿日工作中获得可测量的改善。

## 当前成功标准

- 新 AI 不依赖旧聊天即可正确说明鸿日当前阶段、目标、已批准决定、开放问题和本轮任务。
- 当前状态回答准确，过期方案不会因语义相似而成为当前方向。
- 会议中的 `VIEW`、`PREFERENCE`、`HYPOTHESIS`、`OPTION`、`TENDENCY` 和暂定日期不会被自动升级。
- 新会议只产生增量 Proposal，未经 Fox 确认不覆盖当前状态。
- 每项重要结论可以回到原始文件、会议、发言人、时间和适用范围。
- Codex、Claude 和其他模型读取同一版本的 Task Packet 与项目状态。
- 探索模式能形成不同战略选择及代价；执行模式不擅自重写已批准方向。
- 策略与文案质量通过 Fox 的匿名评审，并优于未使用项目状态层的基线。
- Fox 重复解释背景的次数和耗时有可复核下降。

在线率、并发量、RPO、RTO 和灾备属于团队服务器化后的成功指标，不用于替代当前产品价值验证。

## 产品行为原则

1. AI 按“当前状态 -> 当前阶段与任务 -> 已批准决定 -> 开放问题 -> 相关证据 -> 必要原文”的顺序读取，不从全量文件随机检索开始。
2. AI 只能提出状态变化，不能批准事实、决定、约束、负责人、截止时间或提交版本。
3. 工作模式必须显式选择和切换；AI 不得自行判断信息已经足够并替人收口。
4. 探索协议保留矛盾、发展选择和揭示代价；执行规格严格服从已批准方向、事实与验收标准。
5. 新资料采用增量摄取、冲突检测和变更提案，不静默重写项目历史。
6. 重要判断始终提供证据入口；证据不足时明确标注“未确认”。
7. 不同模型共享项目事实和 Task Packet，但允许在能力、方法和表达上不同。
8. 人类拥有最终判断权，品牌质量不能被结构完整或模型自信替代。

## 产品形态

### 当前候选形态

- **本地工作空间**：以 `/Users/fox/work` 为目标使用路径，保存鸿日原始资料、结构化状态、证据关系和 Proposal。
- **本地应用服务**：向桌面界面、CLI、MCP 和 Skills 提供同一版本化读取与提案入口。
- **本地客户端**：优先评估基于 OpenWork MIT 社区核心改造的 Brand OS Desktop；它是候选客户端基础，不是业务真相源，也不要求先部署 OpenWork Server。
- **多模型入口**：Codex、Claude 和其他模型通过相同 Task Packet、证据协议和输出约束工作。
- **轻量人类界面**：让 Fox 查看当前状态、证据、增量变化、待确认项和 AI 运行，不复制通用项目管理系统。

### 试点后候选形态

以下内容统一标记为 `future-candidate`、`not-approved-for-current-mvp`、`review-after-hongri-pilot`：

- 团队服务器、PostgreSQL、S3/MinIO、OIDC、RLS、Outbox、高可用和灾备；
- 完整团队桌面分发与轻量 Web 后备；
- FoxWork 团队文件协作合并；
- Zvec、Open Notebook、Nubase、FlowLong 和 Dify 的生产集成。

## 品牌人格

克制、可信、敏锐。

表达应直接、准确、有判断力；不以功能数量制造强大感，不以模糊的 AI 自信代替证据，也不把开放问题伪装成确定答案。

## 反例

- 不做以聊天框为唯一入口的通用 RAG 产品。
- 不做换皮的 OpenWork、Codex 或 Claude Code。
- 不做 Jira、Worktile 或 Teambition 式项目管理复制品。
- 不把全部资料直接塞进上下文，也不把向量相似度当作当前有效性。
- 不把工程上的“字段完整”误认为策略上的“问题已经解决”。
- 不在探索模式下强行生成唯一答案、命名和口号。
- 不让 Agent 会话、模型摘要、检索索引或第三方工作流成为项目真相源。
- 不用团队服务器的完整度掩盖个人试点尚未证明价值。

## 可访问性与包容性

- 目标达到 WCAG 2.2 AA。
- 核心工作流支持键盘操作和清晰焦点状态。
- 状态不能只依赖颜色，必须同时使用文字或图标。
- 支持减少动态效果；内容默认可见，不依赖动画加载。
- 中文长标题、长文件名、长原话和中英文混排必须稳定换行且不遮挡。
- 面向 Fox 的界面使用自然业务语言；内部分类码可作为辅助信息，不主导日常界面。
