# Product

## Register

product

## 产品定义

`Brand Project OS` 是当前项目名，最终发行名可以另定。它是一套长期品牌项目的状态与品牌认知协作系统，让员工和不同 AI 在开始工作时都能知道：

- 哪些内容是可回源的事实，哪些只是观点、偏好、假设、选项或倾向；
- 哪些决定和约束已经由人确认；
- 项目现在处于什么阶段，本轮要做什么；
- 当前是在探索、评估、决策还是执行；
- 新会议和资料相对当前状态改变了什么；
- 重要结论来自哪份原件、哪场会议和哪句原话。

它不是普通项目管理软件、通用企业知识库或全量文件向量化 RAG。AI 不替人做最终判断。

## 产品形态

本文用 Brand Project OS 指整套产品，包括员工客户端和公司服务器服务。改名只改品牌标识，不改变产品边界。

| 部分 | 当前决定 |
|:---|:---|
| 第一用户 | Fox |
| 第一验证项目 | 鸿日 |
| 员工客户端 | 基于 OpenWork 的公司定制版；最终发行名待定 |
| 安装数量 | 1；OpenCode Runtime、Sidecar 和本机桥接随包交付 |
| 团队后端 | Brand Project OS Service，部署在公司服务器 |
| 正式状态 | Phase 1 SQLite；切换后 PostgreSQL 唯一写入权威 |
| 原始证据 | Phase 1 本地内容寻址快照；切换后对象存储版本和 SHA-256 |
| AI 接入 | MCP Gateway、Skills、CLI/API |
| 工作流 | Dify 接入；Zvec、Open Notebook、Nubase、FlowLong 逐项评估 |
| 第二客户端 | 不建设 Web/PWA 或另一个桌面客户端 |

MCP 是 AI 接口，不是数据库或业务核心。Desktop 通过业务 API 完成员工操作，其他 Agent 通过受限 MCP/Skills 使用同一应用服务。

FoxWork 的全量团队文件同步、NAS、个人工作区和 PPT 合版仍是独立需求线。品牌项目服务器只接入经授权的资料和状态，不自动接管公司全部文件。

## 产品目的

产品要同时证明两件事：

1. 事实、证据、状态、版本和变更不混乱，重要结论可回源，AI 不能擅自改变正式状态。
2. 系统能保留有价值的不确定性，提出真正不同的策略选择，并在方向获批后稳定执行。

成功不看上传文件数、模型数或组件数。要看真实工作中是否少了重复解释、错用旧方案和人工纠错，同时品牌结果达到人工评审基线。

## 行为原则

1. AI 按“当前状态 -> 当前阶段与任务 -> 已批准决定 -> 开放问题 -> 相关证据 -> 必要原文”的顺序读取。
2. AI、MCP、Skill、Dify、FlowLong 和服务账号只能读取或创建 Proposal，不能批准事实、决定、约束、负责人、日期或外部承诺。
3. 业务确认必须来自有权限员工的交互式客户端，并记录身份、理由、证据、差异和状态版本。
4. 工作模式显式选择和切换。AI 不得自行判断信息已经足够并替人收口。
5. 新资料只产生增量变化、冲突和 Proposal，不静默重写历史。
6. 重要结论提供证据入口；证据不足时明确标记“未确认”。
7. 不同模型共享同一正式状态和 Task Packet，表达和方法可以不同。
8. Tool Permission 只控制本次 Agent 能否调用本机工具，不是业务批准。

## 数据与一致性

- 原件按来源版本和 SHA-256 保存，不以同名文件覆盖旧证据。
- 人工批准事件是状态形成依据，当前投影可从事件重建。
- 正式写入使用幂等键和 `expected_version`；冲突返回差异，不做最后写入覆盖。
- Phase 3 以一次性迁移和写入冻结完成 SQLite 到 PostgreSQL/S3 的切换。
- 服务器接受新正式写入后，本地库只读，不允许回升为第二个权威源。
- OpenWork/OpenCode Session、缓存、索引、摘要、Dify、Zvec、Notebook 和 Memory 都是运行态或派生数据。

## 员工体验

员工打开一个应用即可：

- 查看当前阶段、任务、决定、约束、开放问题和最近变化；
- 打开原始文件或会议片段，核对 AI 依据；
- 查看新会议产生的增量变化和冲突；
- 运行 AI，查看 Task Packet、工具请求、Artifact 和流程状态；
- 逐项批准、修改、驳回或暂缓 Proposal；
- 查看服务、同步、Runtime、备份和版本状态。

聊天是 AI 工作的一种呈现，不是项目首页或正式状态源。服务器不可用时，客户端显示最后水位并进入只读或草稿状态，不把缓存冒充最新事实。

## 成功标准

- 新 AI 不依赖旧聊天即可说明当前阶段、目标、决定、开放问题和任务。
- 过期方案不会因语义相似被当成当前方向。
- `VIEW`、`PREFERENCE`、`HYPOTHESIS`、`OPTION`、`TENDENCY` 和暂定日期不会被自动升级。
- 新会议只产生增量 Proposal，未经员工确认不改变正式状态。
- 每项重要结论可以回到文件、会议、发言人、时间和适用范围。
- Codex、Claude 和其他模型读取相同版本的 Task Packet 与项目状态。
- 多客户端并发不会丢失数据或静默覆盖。
- 服务中断、数据恢复、身份撤权和坏版本回滚有实际演练证据。
- 策略与文案通过匿名人工评审，真实工作中的重复解释和纠错有下降。

## 阶段

1. Phase 0：边界、协议和黄金测试，已完成。
2. Phase 1：公司定制 OpenWork 单一客户端本地纵切，进行中。
3. Phase 2：PostgreSQL/S3、身份权限、一致性、API、审计和恢复。
4. Phase 3：Desktop 联网、远程 MCP/Skills、Dify 和外部组件适配。
5. Phase 4：团队真实工作、并发、故障、安全、容量、签名分发和生产门。

阶段已批准不等于完成。每项功能必须单独实现、测试、记录遥测并提交；阶段门未通过时不进入下一阶段。

## 品牌人格

克制、可信、敏锐。

表达直接、准确、有判断力。不靠功能数量制造强大感，不用模糊的 AI 自信代替证据，也不把开放问题伪装成确定答案。

## 反例

- 不做以聊天框为唯一入口的通用 RAG。
- 不另做第二个员工客户端。
- 不让 OpenWork/OpenCode 会话、模型摘要、检索索引或第三方工作流成为项目真相源。
- 不把 Tool Permission、MCP 令牌或服务账号当作人工身份。
- 不把向量相似度当作当前有效性。
- 不在探索模式强行生成唯一答案、命名和口号。
- 不通过本地与服务器双写解决团队一致性。
- 不以“服务器已启动”代替恢复、权限、并发和真实使用验收。

## 可访问性

- 目标达到 WCAG 2.2 AA。
- 核心工作流支持键盘操作和清晰焦点状态。
- 状态不能只依赖颜色，必须同时使用文字或图标。
- 支持减少动态效果；内容默认可见，不依赖动画加载。
- 中文长标题、长文件名、长原话和中英文混排必须稳定换行且不遮挡。
- 日常界面使用业务语言，不向员工暴露 RLS、Outbox、MCP Token 等开发术语。
